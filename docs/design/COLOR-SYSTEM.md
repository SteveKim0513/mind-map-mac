# 색 관리 체계 (Color System)

> 서비스 전체 색의 단일 기준. 색은 **역할(role) × 형태(form)** 로 관리한다. 색상(hue) 수는 한정돼 있으므로 **형태가 1차 구분 신호**, 색상은 보조.

## 3개의 평면

| 평면 | 토큰 | 형태(form) | 채도 |
|---|---|---|---|
| **구조** | `--ink* / --surface / --canvas / --hairline / --edge` | 배경·텍스트·선 | 무채색 |
| **인터랙션** | `--primary`(파랑) | 선택·포커스·웹링크·활성 | 비비드 |
| **의미 — 상태** | `--state-*` | **아이콘 칩/배지/위성** | **비비드** |
| **의미 — 태그** | `--tag-*` | **노드/섹션 몸체 틴트(아이콘 없음)** | **파스텔(muted)** |

### 핵심 규칙
1. **태그와 상태는 같은 픽셀 역할을 공유하지 않는다.** 태그 = 몸체, 상태 = 칩. 같은 hue여도 **형태 + 채도**로 구분.
2. **하드 예약**:
   - **파랑** = 인터랙션 전용(`--primary`). 태그·상태에 파랑 없음.
   - **빨강** = 삭제(`--state-danger`) 전용. 태그의 'red' 키는 **로즈(rose)** 로 빼서 진짜 빨강과 분리.
3. **상태는 항상 아이콘을 동반**한다(글자색만으로 의미 전달 금지).
4. **태그는 항상 파스텔**이며 노드/섹션 몸체에만 쓴다.

## 상태 색 (Layer 2) — 비비드 아이콘 칩
| 토큰 | 의미 | hue |
|---|---|---|
| `--primary` | 선택·포커스·웹링크 | 파랑 |
| `--state-map` | 마인드맵/노드 정체성 · 노드↔노트 링크 · 피커 | teal |
| `--state-note` | 노트 정체성 · 노트 칩/위성/팝업 | 보라(violet) |
| `--state-scheduled` | 스케줄·미리알림·시간 | 주황(orange) |
| `--state-done` | 완료·저장·진행 | 초록 |
| `--state-danger` | 삭제·파괴적 동작 | 빨강 |
| `--state-search` | 검색 매치 하이라이트 | 골드 |

→ 7개가 서로 다른 hue. 모두 **아이콘 + 비비드**로 등장.

## 태그 팔레트 (Layer 3) — 파스텔 몸체 틴트
사용자가 노드/섹션에 다는 색. **8개 키(`red·orange·yellow·green·teal·violet·pink·brown`)**, 값은 파스텔. 문서엔 **키만 저장**(`theme/palette.ts`) → 값은 언제든 재튜닝 가능.

- 'red' 키 = **로즈**(삭제 빨강과 분리), 파랑 없음.
- 상태칩보다 **한 단계 muted(L↑ S↓)** → 몸체 틴트는 차분, 칩은 또렷.
- 노드 태그: 좌측 바 + 옅은 배경(`color-mix(tag 13%, surface)`).
- 섹션 태그: 같은 팔레트, 큰 gooey 영역 → 형태·스케일로 노드 태그와 구분.

## 왜 같은 hue 공유가 괜찮은가
태그 violet(파스텔 몸체) vs 노트 칩 violet(비비드 + 아이콘)은 **형태와 채도**가 달라 한눈에 갈린다. 14개 색을 전부 다른 hue로 만들면 오히려 서로 안 구분되므로, **형태 우선 + 채도 분리 + 파랑/빨강 예약**이 가장 견고하다.

## 변경 이력 시 체크리스트
- 새 상태색 추가? → 반드시 아이콘 동반, 기존 상태/primary와 hue 충돌 확인.
- 태그 추가/튜닝? → 파스텔 유지, 파랑·삭제빨강 금지, 서로 구분되는지 확인.
- 절대 토큰을 한 곳(이 문서 + `:root`)에서만 정의. 컴포넌트에 raw hex 금지.
