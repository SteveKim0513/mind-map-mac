# 0010 — 노트 이미지 첨부 폴더를 진짜 숨김 폴더로 (`notename.assets/` → `.notename.assets/`)

> 날짜: 2026-07-14 · 상태: 채택

## 맥락

노트에 이미지를 넣으면(v0.7.15 이후) `notename.assets/` 폴더에 파일로 저장된다(base64 인라인 대신 외부 파일 — `docs/exec-plans/completed/2026-06-23-note-image-external-storage.md`). 이 폴더는 앱 사이드바 트리 스캐너(`electron/main.ts`의 `walk()`)에서는 이름이 `.assets`로 끝난다는 이유로 걸러져 안 보이지만, 이름 앞에 점이 없어 **Finder 등 앱 밖에서는 평범하게 보이는 폴더**다. 사용자가 "숨김 폴더인 줄 알았는데 보인다"고 지적했다.

이미지 첨부 폴더는 [0009 — 노드 노트 위치](0009-node-note-location.md)가 숨김 처리를 폐기한 "노트 파일"과는 다른 대상이다. 0009는 **사용자가 만든 콘텐츠**(찾아야 하는 노트)를 숨기면 안 된다는 결정이었고, 이번 대상은 **내부 저장소**(이미지는 항상 노트 본문에 임베드되어 보이므로, 폴더 자체를 사용자가 직접 열어볼 일이 없다)다. 0009와 모순되지 않는다.

## 결정

1. **폴더명을 `.notename.assets/`(점으로 시작)로 바꾼다.** macOS/Finder의 표준 숨김 규칙을 그대로 타므로 앱 밖에서도 안 보인다.
2. **기존(점 없는) 폴더도 계속 정상 동작한다** — `findAssetsDir()`가 두 가지 이름을 모두 인식한다. 새 이미지를 추가로 넣으면 기존 폴더를 재사용(분할 방지)하고, **이름 변경·이동·휴지통 이동·복원처럼 폴더를 이미 다른 곳으로 옮겨야 하는 시점에 한해** 숨김 이름으로 승격(`relocateAssetsDir()`)하면서 본문의 이미지 참조도 새 이름으로 다시 쓴다.
3. **일괄 마이그레이션은 하지 않는다.** 워크스페이스를 열 때 모든 노트를 훑어 폴더명을 한 번에 바꾸는 방식은 고려했으나 기각했다 — 이미 열려 있는 탭이 마이그레이션 전 상태의 본문을 메모리에 들고 있다가 자동저장(800ms 디바운스)으로 그걸 그대로 덮어써 버리면, 폴더는 이미 새 이름으로 바뀐 뒤라 이미지 참조가 깨질 위험이 있다. 반면 이름 변경/이동/휴지통 처리 시점은 이미 그 순간에 본문을 읽어 참조를 다시 쓰는 기존 로직(v0.8.1에서 고친 리네임 버그)이 있어 같은 안전한 경로에 얹을 수 있다.

## 결과

- 새로 만들어지는 모든 이미지 첨부 폴더는 즉시 진짜 숨김 폴더다.
- v0.8.3 이전에 만들어진 폴더는 해당 노트를 건드리기 전까지 계속 보이는 상태로 남는다 — 사용자가 굳이 정리하고 싶다면 노트 이름을 한 번 바꾸는 것으로 충분하다(그 순간 자동 승격).
- 다시 볼 조건: 사용자 다수가 "여전히 옛날 폴더가 보인다"고 불편해하면, 0009 §3처럼 워크스페이스 로드 시점의 명시적 일괄 마이그레이션을 추가하는 것을 재검토한다 — 이번엔 자동저장과의 경합을 피하기 위해 마이그레이션이 끝난 뒤에만 탭 복원이 노트 내용을 읽도록 순서를 보장하는 설계가 먼저 필요하다.
