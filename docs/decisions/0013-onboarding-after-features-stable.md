# 0013. 온보딩(첫 실행 코치)은 기능이 안정된 뒤에 도입한다

날짜: 2026-07-16 · 상태: 채택

## 맥락 — 어떤 문제/선택지가 있었나

v0.10.0에 첫 실행 코치(`FirstRunCoach` — 빈 맵에서 "Enter로 적기 → Tab으로 잇기 → @내일 3시로 일정"을
안내)가 함께 도입됐다. 그러나 이 프로젝트의 방향은 **"온보딩은 기능이 다 fix된 뒤에 만든다"**였다(사용자
지시). 지금은 캘린더·일정·집중·접근점 동선이 빠르게 바뀌는 중이라, 이 시점에 만든 온보딩은 곧 낡는다
(가리키는 단축키·동선이 바뀌면 안내가 틀려짐). 즉 조기 온보딩은 사용자 가치보다 유지보수 부담이 크다.

## 결정 — 무엇을 택했나

**핵심 기능이 안정화되기 전에는 온보딩/코치 UI를 만들지 않는다.**

- v0.10.0에 조기 도입된 `FirstRunCoach`를 **제거**한다(컴포넌트·`.frc` CSS·`Pane.tsx` 연결·`onboardingSeen`
  localStorage·전용 E2E). 빈 맵에는 기존의 조용한 빈 상태 안내(`Enter`로 중심 주제 시작)만 둔다.
- 온보딩은 맵·노트·일정·집중·캘린더의 동선이 **확정된 뒤** 별도 작업으로 설계한다.

## 결과 — 트레이드오프, 다시 볼 조건

- 장점: 곧 낡을 온보딩의 유지보수를 제거. 기능 변경이 온보딩을 깨뜨릴 걱정 없음. 빈 상태 힌트는 유지되어
  첫 사용자도 `Enter`로 바로 시작할 수 있다.
- 트레이드오프: 신규 사용자용 안내가 최소한(빈 상태 힌트 + 설정 안 "사용 안내" 매뉴얼)에 그친다. 기능
  확정 후 제대로 설계해 보완한다.
- **다시 볼 조건**: 핵심 기능 동선이 안정됐다고 판단되면 온보딩을 재설계·재도입(그때 이 결정을 갱신).
- **재발 방지**: "기능 안정화 전 온보딩/코치 금지"를 이 기록으로 남긴다. 온보딩 관련 작업 제안 시 먼저 이
  결정을 확인한다.
