# 0015. 집중 게이트 해제 — 일정과 집중은 독립 기능

날짜: 2026-07-16
상태: accepted

## 컨텍스트

[결정 0011](0011-three-axis-thinking-model.md) §3은 "집중 세션은 일정(scheduled) 노드에서만 시작"으로 게이트를 걸었다 — 집중이 core(일정)를 실행하는 모드라는 정의. [결정 0014](0014-todo-node.md)는 완료·일정·집중을 할 일(todo) 노드 전용으로 재편하면서 "일정 있으면 집중"을 유지했다.

실사용에서 사용자는 "일정 노드로 전환 → 스케줄 지정 → 집중"의 **순차 종속**이 불편했다. 일정을 잡지 않아도 지금 이 할 일에 바로 집중하고 싶다 — 일정 지정과 집중은 서로 다른 행위(계획 vs 실행 착수)다.

## 결정

**집중을 일정에서 분리한다.** 할 일(todo) 노드에서는 **완료 · 일정 지정 · 집중이 각각 독립 기능**으로 노출된다. 집중은 더 이상 `scheduleAt`/`scheduled`를 요구하지 않는다. **todo 게이트(0014)는 유지** — 일반 노드(순수 생각)에는 여전히 실행 UI가 없고 "할 일로 전환"만 보인다.

- `requestFocusStart`(`focus/controller.ts`)는 원래 일정 가드가 없었으므로 컨트롤러 변경 없음.
- 게이트는 UI 3곳에만 있었다 → 조건 제거: `SelectionToolbar`(집중 버튼의 `node.scheduled` 조건 제거 + "일정 지정" 버튼 추가), `ContextMenu`(집중 항목의 `disabled={!node.scheduled}` 제거). `SchedulePopover`의 "지금 집중 시작"은 일정 편집 맥락이라 그대로 둔다.

## 결과 (장단점)

- **장점**: "할 일로 전환"만 하면 일정·집중을 곧바로 쓸 수 있어 실행 진입 마찰이 준다. 계획(일정)과 실행 착수(집중)가 개념적으로 분리된다. 클릭 팝업(SelectionToolbar)이 더는 막다른 길이 아니다(일정 지정 입구가 생김).
- **단점**: 0011 §3의 "집중 = 일정을 실행하는 모드"라는 서사를 완화한다. 집중이 일정 없이도 가능해져 "계획된 것을 실행"이라는 흐름이 약해질 수 있다. **완화**: todo 게이트로 여전히 '실행 대상'으로의 명시적 전환은 필요하므로, 순수 생각과 실행의 경계는 유지된다.

## 대안

- **게이트 유지**: 기각 — 사용자 마찰(일정 없이는 집중 불가).
- **아무 노드나 집중(todo 무관)**: 기각 — 0014의 "생각/실행 분리"를 훼손.
