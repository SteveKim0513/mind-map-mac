# 자연어 스케줄·태그 파싱 (노드 텍스트 커밋 시)
날짜: 2026-07-10
상태: completed

> REDESIGN-VISION-2026-07.md §3-2 (A1). 3단계(고위험) 항목 — "오탐 방지 룰을 먼저 설계"하라는
> 문서 자체의 요구에 따라 구현 전 이 계획을 작성한다.

## 목표

노드 텍스트를 커밋할 때(Enter/blur) "내일 3시", "다음 주 월요일" 같은 명확한 패턴을 감지해
`scheduled`+`scheduleAt`을 자동으로 채우고, `#태그명` 토큰을 감지해 `color`를 자동 지정한다.
SchedulePopover와 우클릭 색 그리드는 그대로 남아 "자동 인식이 틀렸을 때 고치는 보조 경로"가 된다.

## 범위 (포함)

- 순수 함수 `parseScheduleText(text, now)` — 요일·상대날짜·시각 조합만 인식.
- 순수 함수 `parseHashtagColor(text)` — `#색상명` 토큰(한국어 별칭 + 영어 키) 인식.
- `mapStore.ts`의 `commitText`가 텍스트 커밋과 같은 draft mutation 안에서 두 파서 결과를 반영
  (한 번의 Enter = 한 번의 undo 스텝 유지 — `setScheduleAt`/`setColor`를 별도로 호출하면 undo가
  여러 스텝으로 쪼개지므로, 필드를 draft에 직접 대입).
- 인식된 텍스트는 **그대로 둔다**(안 잘라냄) — 이미 존재하는 스케줄 칩(`.gchip.sched`)과 태그
  틴트가 "인식됐다"는 시각적 신호를 제공하므로, 별도의 인라인 하이라이트 렌더링(에디터 내부
  substring 강조)은 v1 범위에서 제외한다.

## 범위 (제외 — 위험도/공수 대비 낮은 가치)

- 인라인 텍스트 하이라이트(옅은 배경으로 인식된 부분만 마킹) — contentEditable range 조작이
  필요해 공수·위험이 큼. 기존 스케줄 칩/색 틴트로 "인식됐다" 신호는 이미 충분하다고 판단.
  필요성이 실사용에서 확인되면 별도 exec-plan으로 추후 진행.
- 절대 날짜(예: "7월 15일", "2026-08-01") — v1은 상대날짜(오늘/내일/모레/글피/다음 주 요일)만.
- 반복 일정, 시간 범위("3시~5시") — 미지원.
- 텍스트 중간의 애매한 패턴 학습/제안 UI — 파싱은 침묵하거나 확정 적용만, 중간 상태 없음.

## 오탐 방지 룰 (핵심 — REDESIGN-VISION §5 위험 항목 대응)

**일정 파싱은 아래 두 요소가 모두 있어야 매칭한다: (날짜 표현) 또는 (요일 표현), 그리고 선택적
시각 표현.** 날짜/요일 표현 없이 시각만 있는 경우("3시에 만나") 오늘 날짜로 매칭하지 않는다 —
"3시"만으로 스케줄 의도인지 본문 서술인지 구분 불가하므로 무반응이 안전.

- 상대날짜: `오늘|내일|모레|글피` (정확히 이 4단어만, 접두/접미 문자 없이 단어 경계)
- 요일: `(다음\s?주\s?)?(월|화|수|목|금|토|일)요일` — "다음 주"가 있으면 +7일, 없으면 이번 주
  해당 요일(이미 지났으면 다음 주로 자동 보정 — SchedulePopover의 `nextWeekend()`와 동일 원칙).
- 시각(선택): `(오전|오후)?\s?(\d{1,2})시(\s?(\d{1,2})분)?` — 오후 명시 시 +12시간(12시 제외),
  오전/오후 없이 1~7시는 오후로 추정하지 않는다(모호하면 24시간제 그대로, 즉 "3시"=03:00).
  시각이 없으면 09:00 기본값(SchedulePopover 빠른 칩 "아침 09:00"과 동일 관례).
- 날짜/요일 표현이 전혀 없으면 어떤 경우에도 매칭하지 않는다 — 숫자·시각 표현 단독으로는
  스케줄 의도로 해석하지 않는 것이 이 룰의 핵심(문서 §5: "과탐지보다 무반응이 안전").

**색상 파싱**: `#` 다음에 오는 한글/영문 토큰이 `TAG_KEYS`(red·orange·yellow·green·teal·violet·
pink·brown) 또는 그 한국어 별칭(빨강·주황·노랑·초록·청록·보라·분홍·갈색)과 정확히 일치할 때만
매칭. 일치하지 않는 `#아무거나`는 무시(에러 없음, 텍스트에 그대로 남음 — 임의 해시태그 기능은
범위 밖).

## 검증 방법

- 단위: `parseNodeText.test.ts` — 매칭 케이스(요일/상대날짜×시각 유무) + 오탐 방지 케이스
  ("내일 회의 있음"→매칭[의도적, 날짜 표현 있음]/"3시에 밥"→불매칭[시각만] 등 경계 케이스).
- E2E: 노드에 "내일 3시"를 입력 후 커밋 → `.gchip.sched` 노출 확인. `#red` 입력 → 노드 색 태그
  반영 확인. 오탐 방지 케이스("할 일이 3개 남았다")가 `scheduled` class를 얻지 않는지 확인.
  (단일 undo 스텝은 `commit()` 호출이 하나임을 코드로 보장 — ⌘Z가 네이티브 Electron 메뉴
  가속기로만 연결돼 있어 Playwright 합성 keydown으로는 E2E 재현이 불가능해 별도 검증 안 함.)
- `make verify`, `make dev-safe` 육안 확인.

## 발견한 사실 (작업 중 갱신)

- "다음 주 X요일" 계산에서 초기 구현("이번 주 delta" + 무조건 +7)이 이중 계산 버그를 냈다 — 이번 주
  해당 요일이 이미 지난 경우 "요일만"과 "다음 주 요일"이 같은 날짜로 수렴해야 하는데 +7이 한 번
  더 붙어 한 주 더 밀렸다. 이번 주 일요일을 앵커로 삼아 목표 요일까지의 오프셋을 계산하고,
  `다음 주` 플래그가 있을 때만 +7을 더하는 방식으로 수정(단위 테스트로 회귀 방지, `parseNodeText.test.ts`).
- `commitText` 안에서 `!n.scheduled`/`!n.color` 가드를 추가 — 이미 수동으로 일정·색을 지정한
  노드를 나중에 다시 편집해도 자동 파싱이 덮어쓰지 않는다(SchedulePopover/색 그리드가 "보조
  경로"로 격하된다는 문서 취지를 지키기 위한 실무적 보완).
- ⌘Z(undo)는 네이티브 Electron 메뉴 가속기로만 연결돼 있어 Playwright의 합성 keydown으로는
  재현이 안 됨 — 단일 undo 스텝 보장은 코드 리뷰(하나의 `commit()` 호출)로만 검증, E2E 생략.
- 실제 구동 확인(`make dev-safe` 대체용 Playwright 스크린샷): "내일 오후 3시 디자인 리뷰 #보라"
  입력 시 일정 칩("내일 15:00")과 보라색 틴트가 동시에 붙는 것, "할 일이 3개 남았다"(오탐 후보)는
  아무 것도 안 붙는 것을 라이트 모드에서 육안 확인.
