# 전역 즉시 캡처 (Global Quick Capture)
날짜: 2026-07-10
상태: completed

> REDESIGN-VISION-2026-07.md §3-1 (A1/A4). 3단계(고위험) 항목 — "OS 권한/다른 앱과의 충돌 검토
> 먼저"라는 문서 자체의 요구에 따라 구현 전 이 계획을 작성한다.

## 목표

앱이 백그라운드에 있거나 메인 창이 닫혀 있어도, 전역 단축키 한 번으로 작은 캡처 입력창이 뜨고
텍스트를 입력해 Enter를 누르면 워크스페이스의 고정 맵("오늘의 생각.mind")에 새 루트 노드로
즉시 쌓인다 — Apple Notes의 Quick Note(Fn+Q)/Reminders 빠른 추가와 동일한 마찰-없는 캡처.

## 조사 결과 (구현 전 확인)

- **macOS 권한**: Electron `globalShortcut.register()`는 Accessibility/입력 모니터링 권한이
  필요 없다(입력 시뮬레이션이나 다른 앱 UI 읽기가 아니라 OS가 특정 키 조합을 이 앱에 라우팅하는
  것뿐). 따라서 AGENTS.md의 "새 macOS 시스템 권한 요청 시 인간 승인 필요" 규칙에는 걸리지 않는다.
  단, 이 앱이 처음으로 "포그라운드 없이 백그라운드에서 항상 떠 있는" 동작을 갖게 된다는 점은
  사용자 기대치를 바꾸는 변화라 명시적으로 알린다.
- **앱 생존 여부**: `electron/main.ts:189` `app.on('window-all-closed')`가 macOS에서는 이미
  `app.quit()`를 안 부른다(`process.platform !== 'darwin'` 조건) — 즉 메인 창을 닫아도 앱
  프로세스는 Dock에 남아 있다. 전역 단축키 등록도 이 생존 기간 동안 계속 살아있다. 앱을
  완전히 종료(⌘Q)하면 당연히 캡처도 꺼진다 — 이건 트레이 상주 앱으로 바꾸지 않는 한 피할 수
  없는 제약이고, 이번 범위에서는 트레이 아이콘 추가는 하지 않는다(범위 밖 — 아래 참조).

## 범위 (포함)

1. **단축키**: `Alt+Space`(⌥Space) 하나만 고정 바인딩(설정 UI로 변경 가능한 옵션은 v1 제외).
   충돌 시 등록 실패를 감지해 조용히 무시(앱이 죽지 않게) + 설정의 "고급 설정"에 상태 표시
   (등록 성공/실패, 실패 시 "다른 앱이 이 단축키를 사용 중이에요" 안내).
2. **캡처 창**: 별도의 작고 테두리 없는 `BrowserWindow`(`alwaysOnTop`, `frame:false`,
   `resizable:false`, 화면 중앙 근처, 약 480×72px) — 렌더러는 기존 앱 번들을 `?capture=1` 쿼리로
   로드해 `App.tsx`가 이 경우 전체 앱 대신 `CaptureWindow.tsx`(입력 하나 + 안내 문구)만 그린다.
   메인 프로세스가 아닌 렌더러가 `io/formats.ts`를 통해 문서 스키마를 다루도록 해 스키마 지식이
   `electron/main.ts`에 새로 생기지 않게 한다(아키텍처 규칙 — 문서 스키마는 `src/io/`가 유일한
   소스).
3. **저장 대상**: 워크스페이스 루트의 `오늘의 생각.mind`(사용자가 만든 일반 파일 — 시스템 폴더
   아님, 사이드바에 정상적으로 보임). 없으면 새로 생성. Enter 시 새 루트 노드를 추가하고 즉시
   파일에 저장, 입력창을 비우고 창을 숨긴다(닫지 않고 숨김 — 다음 호출 시 재생성 비용 절감).
   Escape는 저장 없이 숨김.
4. **새 IPC**: `capture:show`(단축키 → 메인이 캡처 창 표시/포커스, 렌더러 호출 아님, 메인
   내부에서만 사용) / `capture:append(text)`(렌더러 → 메인: 워크스페이스 루트의
   "오늘의 생각.mind"에 새 루트 노드 텍스트를 append 요청 — 단, 실제 JSON 파싱/직렬화는 렌더러가
   `io/formats.ts`로 하고, 메인은 `fs:read`/`file:save` 기존 채널로 충분하므로 **캡처 전용 새
   채널은 파일 경로 확보(`capture:targetPath`)만 추가**하고 나머지는 기존 `readFile`/`save`
   IPC를 재사용한다.

## 범위 (제외)

- 단축키 커스터마이징 UI(설정에서 키 조합 변경) — v1은 고정, 충돌 시 안내만.
- 시스템 트레이 상주 아이콘·앱 완전 종료 후에도 동작 — Dock에서 완전히 나가면(⌘Q) 캡처도 꺼짐.
- 캡처 창에서 태그·일정·색 지정(3-2의 자연어 파싱은 캡처 텍스트에도 동일하게 적용돼 자동으로
  붙는다 — 이건 재사용이지 새 기능 아님).
- 여러 워크스페이스 중 선택 — 항상 현재 열린(마지막) 워크스페이스 기준.
- 캡처 결과를 나중에 다른 맵으로 옮기는 전용 UI(사이드바에서 일반 파일처럼 드래그해서 옮기면 됨).

## 위험 및 완화

- **단축키 충돌**: `globalShortcut.register()`가 `false`를 반환하면(이미 다른 앱이 선점)
  조용히 실패 처리 + 설정 화면에 상태 노출. 앱을 막지 않는다.
- **워크스페이스 미설정 상태에서 호출**: `getWorkspace()`가 이미 기본 워크스페이스를
  보장하므로(없으면 생성) 항상 유효한 경로가 있다.
- **자동 저장 충돌**: "오늘의 생각.mind"가 메인 창에서도 동시에 열려 있는 상태에서 캡처하면
  두 프로세스가 같은 파일에 쓸 수 있다(마지막 쓰기가 이김) — v1은 이 경쟁을 감내한다(발생
  빈도 낮음: 캡처의 목적 자체가 "메인 창을 안 보고 빠르게 적기"라 동시 편집 시나리오가 드묾).
  실사용에서 문제로 확인되면 파일 잠금 또는 IPC 브로드캐스트로 후속 대응.

## 검증 방법

- 단위: 캡처 창 렌더러가 새 루트 노드를 만드는 로직(순수 함수화 가능한 부분)에 대한 테스트.
- E2E: Playwright는 전역 OS 단축키(등록된 accelerator)를 직접 누를 수 없으므로, `app.evaluate`로
  `globalShortcut`이 등록됐는지 확인 + `capture:append` IPC를 직접 호출해 파일에 루트 노드가
  추가되는지 확인(단축키 자체의 종단 간 테스트는 수동 `make dev-safe` 확인으로 보완).
- `make dev-safe`에서 실제 ⌥Space를 눌러 캡처 창이 뜨는지, 텍스트 입력 후 Enter로 사이드바에
  "오늘의 생각" 파일이 생기고 노드가 쌓이는지 육안 확인.

## 발견한 사실 (작업 중 갱신)

- 계획대로 `electron/main.ts`는 `.mind` 스키마를 전혀 모른다 — 캡처 창(렌더러)이
  `io/formats.ts`의 `deserialize`/`serialize`로 읽고 쓰고, 메인은 `capture:targetPath`로 경로만
  알려주고 기존 `fs:read`/`file:save` 채널을 그대로 재사용했다. 새 채널은
  `capture:show`/`capture:targetPath`/`capture:hide`/`capture:status` 4개뿐.
- **`capture:show`를 렌더러에서도 호출 가능한 IPC로 노출한 것**이 계획에는 없었지만 두 가지
  이유로 추가했다: (1) Playwright는 진짜 OS 전역 단축키를 누를 수 없어 E2E가 불가능했는데, 같은
  코드 경로(`showCaptureWindow()`)를 타는 IPC 호출로 우회해 실제 창 생성·저장까지 종단 간
  테스트할 수 있었다. (2) ⌥Space를 몰라도 ⌘K "빠른 캡처 열기" 명령으로 같은 기능을 쓸 수 있어
  발견성이 좋아진다(A5의 "⌘K가 진짜 앞문" 원칙과도 맞음) — 사실상 공짜 보너스.
  즉 이 채널은 테스트 우회로 시작해 정식 진입점으로 승격됐다.
  또한 3-7의 자동 학습(Quick Keys) 대상이 되므로 반복 사용하면 ⌥1~9도 배정된다.
- 캡처 창은 숨김/재표시만 하고 파괴하지 않는다(`hide()`, `closed`에서만 null화) — 두 번째
  이후 캡처는 새 `window` 이벤트가 안 뜨므로 E2E에서 같은 Page를 재사용해야 했다(테스트
  작성 중 처음엔 `waitForEvent('window')`가 두 번째 호출에서 영원히 걸려 타임아웃 — 수정).
- 3-2의 자연어 파싱(`parseScheduleText`/`parseHashtagColor`)을 캡처 텍스트에도 그대로 적용해
  "내일 오후 2시 팀 회고 #teal"을 캡처하면 맵에 직접 입력한 것과 동일하게 일정+색이 자동으로
  붙는다 — 새 로직 없이 기존 순수 함수 재사용.
- `transparent: true` + `frame: false` BrowserWindow는 기본 배경이 불투명해서 `body.capture-mode
  { background: transparent }` CSS를 별도로 추가해야 실제로 투명하게 비쳤다.
- 실제 구동 확인(Playwright 스크린샷): 라이트/다크 모두 캡처 창이 둥근 알약 형태로 올바르게
  렌더링되고, 설정 → 고급 설정에 "전역 캡처 — Alt+Space" 상태 줄이 표시됨을 육안 확인.

## 결정 변경 이력

- `capture:show` IPC 채널: 계획에는 없었으나 테스트 가능성 + 발견성 개선을 위해 추가(위 항목
  참조). 위험도 변화 없음 — 기존 `showCaptureWindow()` 로직을 그대로 노출한 것뿐.
