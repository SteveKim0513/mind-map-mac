## 가정

- `usePopover`류 공유 프리미티브는 `src/ui/`에 신설 — 여러 도메인(canvas/menu/inspector/panes/sidebar)이
  공유해야 하므로 `.claude/rules/architecture.md`상 store를 통하거나 공용 ui 모듈을 통해 연결하는
  것이 맞다. 새 도메인 모듈 생성에 준하므로 이 실행 계획으로 사전 계획 요건을 충족한다.
- 기존 `ContextMenu`/`SchedulePopover`/`NodePopover`/`LinkAddPopover`의 위치 계산·바깥클릭·Escape
  로직을 그대로 프리미티브로 승격하고, 각 컴포넌트는 그 프리미티브를 사용하도록 리팩터링한다
  (동작 자체를 바꾸지 않고 구현을 통합 — 회귀 위험 최소화).

## 위험

- 전략 A(공유 프리미티브 전환)가 가장 위험도가 높다 — 여러 팝오버의 위치 계산·닫기 로직을 하나로
  합치는 과정에서 기존 E2E(스케줄·색상·링크·노드 팝업 관련)가 깨질 수 있다. 각 컴포넌트 전환마다
  개별 검증(`make test`, 관련 E2E) 후 다음으로 넘어간다.
- 전략 D(커맨드 팔레트 노드 동작)는 선택된 노드가 없을 때의 처리, 팔레트에서 실행 후 포커스 복귀
  등 엣지 케이스가 있다 — 각 동작은 기존 SelectionToolbar/ContextMenu 핸들러를 그대로 재사용해
  로직 중복을 피한다.
- 전략 C(진입점 단일화)는 기존 사용자가 익숙해진 진입점(툴바 아이콘)을 없애는 것이므로, "완전히
  제거"보다는 "같은 아이콘/같은 결과"로 통일하는 쪽을 우선한다.

## 검증 방법

각 단계마다: `make verify` → 관련 유닛/E2E 테스트 → `make harness-check`. 전체 완료 후
`make pre-release` 전체 통과 확인 후 배포.

## 발견한 사실 (작업 중 갱신)

- **환경 flake 2건 발견 및 원인 격리 완료** — `e2e/file-management.spec.ts`의 "window regains focus"
  테스트와 `e2e/responsive-toolbar.spec.ts`의 좁은 폭 테스트가 이 세션 후반부터 실패하기 시작했다.
  둘 다 `app.evaluate(({ BrowserWindow }) => ...)`로 실제 창을 `blur()`/`focus()`/`setSize()`하는
  테스트다. `git stash`로 이 세션의 모든 코드 변경을 되돌린 뒤 재실행해도 동일하게 실패해, 이
  세션의 어떤 코드 변경과도 무관한 샌드박스 환경의 창 제어 API 문제로 확인했다(아마 장시간 세션
  동안 누적된 Electron 인스턴스/디스플레이 상태 때문으로 추정). 코드 문제가 아니므로 수정하지
  않았다 — 다음 세션에서 환경이 정상화됐는지 재확인 권장.
