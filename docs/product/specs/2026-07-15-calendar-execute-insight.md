# 캘린더 강화 — 실행하는 캘린더 + 계획↔실행 대조

> 작성: 2026-07-15 · 상태: **구현 완료** · 출처: 사용자 요청("캘린더를 파워풀하게, 우리 철학에 맞게").
>
> **구현 노트(편차)**: (1) 빈 셀 캡처는 캘린더 탭이 활성이라 `activeStore()`가 null이므로 **첫 열린 맵 탭**으로 폴백한다. (2) 월간 셀 클릭은 캡처가 아니라 **일간 이동**(v1 계약·E2E 유지) — 캡처는 주간 그리드 빈 슬롯 전용. (3) 주간 드롭 의미: 그리드=날짜+시각, 종일 띠=종일화, 월간 셀=날짜만. (4) "overdue+집중0" 개별 마커는 별도로 두지 않고 **일간 요약 + 기존 overdue 빨강 틴트**로 대체. (5) 닫힌 맵 드래그는 맵 열기로 강등. 검증: 단위 175 + E2E(전체 68, calendar-execute 5개 신규).
> 기반: [2026-07-15-calendar-view.md](2026-07-15-calendar-view.md)(캘린더 v1 — 읽기 전용 투영). 이 명세는 v1을
> 대체하지 않고 **편집·통찰 표면으로 확장**한다. 방향 근거: [결정 0011](../../decisions/0011-three-axis-thinking-model.md)
> (정리→실행→통찰), [결정 0007](../../decisions/0007-focus-positioning.md)(G3 리트머스).
> 범위 결정: Phase 1(실행)+2(통찰)만 이번 명세. 타임블로킹(duration)·반복은 스키마 변경 → 별도 트랙.

## 1. 문제

캘린더 v1은 **읽기 전용**이다. `collectAgenda`가 모은 `scheduleAt` 노드를 날짜 그리드에 얹어 "보여주기만" 한다.
두 가지가 빈다.

1. **실행이 안 된다.** "이건 내일로 미루자"를 캘린더에서 못 한다 — 노드를 찾아가 SchedulePopover를 열어야
   한다. 캘린더가 시간 축인데 시간을 캘린더에서 바꿀 수 없다.
2. **계획만 보이고 실행이 안 보인다.** 캘린더는 "무엇을 하기로 했나"(`scheduleAt`)만 보여주고, "실제로 얼마나
   했나"(`FocusSession` 시간)는 돌아보기 대시보드에 따로 있다. 우리는 한 노드에 계획·실행·기록을 **동시에**
   소유하는데도, 계획과 실행이 다른 화면에 흩어져 있어 그 간극이 안 보인다.

## 2. 해결

캘린더를 "읽는 화면"에서 **"실행하고 되돌아보는 화면"**으로 승격한다. 스키마 변경 없이, 기존 노드 필드와
집중 세션 데이터만으로 한다.

### Phase 1 — 실행하는 캘린더 (읽기 전용 → 편집)

**(1) 드래그 리스케줄** — v1이 "별도 후속"으로 미룬 항목.
- 월간: 항목 칩을 다른 날짜 셀로 드래그 → 날짜만 변경(시각·종일 여부는 유지).
- 주간: 다른 날짜 칸으로 드래그 → 날짜 변경. 시간 그리드(아래)가 생기므로 세로 드래그 → 시각 변경.
- **반드시 `mapStore.setScheduleAt` 경유.** `reminderOn`이면 리마인더 due도 함께 이동한다(불변조건 보존).
  캘린더가 sync 로직을 우회해 `scheduleAt`을 직접 쓰면 안 된다.
- **열린 맵의 노드만 드래그 가능.** 닫힌 맵(디스크 스캔으로만 agenda에 뜬 노드)은 v1처럼 읽기 전용 —
  드래그하면 "맵을 먼저 여세요" 안내. (닫힌 맵 백그라운드 패치는 후속.)

**(2) 빈 셀 클릭 → 즉시 캡처.**
- 월간/주간 빈 셀(또는 주간 시간대) 클릭 → 인라인 입력창 → Enter 시 그 날짜·시각으로 일정 노드 생성.
- **노드는 맵에 산다** — 활성 맵 탭이 있으면 그 맵에 생성(선택 노드 아래, 없으면 루트 아래). 활성 맵 탭이
  없으면 "먼저 맵을 여세요" 안내. 캘린더가 맵 없는 고아 노드를 만들지 않는다(우리 모델 보존).

**(3) 주간 시간 그리드.**
- 지금 주간은 시각 무관 칩 나열 → **시간대 세로축에 배치**(hasTime 항목은 시각 위치에, 종일은 상단 종일 띠).
- 종일/시각지정 구분은 기존 `hasTime`(scheduleAt의 시각이 00:00이면 종일) 로직 재사용.

**(4) 주/월 뷰 인라인 집중·완료.** 일간엔 이미 있는 `requestFocusStart`/완료 토글을 주·월 항목에도.

### Phase 2 — 계획↔실행 대조 (통찰) ★ 핵심 차별화

우리만 가진 데이터(같은 노드의 계획 + 실제 집중 시간)로 캘린더 위에서 둘을 겹친다. 순수 계산이다.

**(1) 셀에 실제 집중 시간 오버레이.** [focus/aggregate.ts](../../../src/focus/aggregate.ts)의 `dailyTotals`(dayKey→총
집중초) 재사용.
- 월간 셀: 계획 개수 배지 + 그 날 총 집중 시간(있을 때).
- 주간: 각 날 하단에 집중 총량 미니 막대.
- 일간 헤더: **"계획 N · 집중 1h20m · 완료 M"** 한 줄 요약(원칙 2 "상태는 1초 안에 읽힌다").

**(2) 미실행 가시화 → 재계획 동선.**
- overdue이면서 집중 0인 항목을 시각적으로 구분(기존 [scheduleInfo](../../../src/canvas/scheduleInfo.ts)의
  `over` 상태 재사용, **새 색 도입 없음**).
- 그 항목을 Phase 1의 드래그로 오늘·미래로 바로 재배치 → "계획했지만 못 함 → 다시 계획"이 한 화면에서.

**(3) 돌아보기 연결.** 캘린더 헤더의 "돌아보기" 버튼(v1에 이미 있음)에서 해당 기간의 대시보드로 — 실행을
보다가 통찰로 넘어가는 동선을 완성.

## 3. 데이터·아키텍처 (스키마 무변경 확인)

- **새 필드 없음. `MindMapDoc.version` 그대로 → 인간 승인 불필요.** 계획=기존 `scheduleAt`, 실행=기존
  `FocusSession`.
- `scheduleAt`은 **로컬 벽시계 ISO**(오프셋 없음). 드래그 날짜 계산은 전부 로컬 `Date`로 — **UTC 금지**
  (섞이면 리마인더 라운드트립이 깨진다).
- 모든 일정 뮤테이션은 `mapStore` 액션(`setScheduleAt`/`setScheduled`) 경유 → `reminderOn/reminderId`
  불변조건과 고아 정리(`reminderDeleteHook`)가 자동 보존.
- 아키텍처 경계: `src/calendar/`는 `store/`·`focus/`만 참조(기존 격리 유지). 드래그/생성 UI는 `calendar/`,
  날짜 계산은 [calendarMath.ts](../../../src/calendar/calendarMath.ts) 순수 함수로 확장.

## 4. 범위에서 뺀 것 (Phase 1+2)

- **타임블로킹(`durationMin` 블록 길이)** — 스키마 필드 추가 → Phase 3, 인간 승인 트랙.
- **반복 일정** — `version` 스키마 변경 + 마이그레이션 + 리마인더 반복 동기화 → Phase 4, 별도 ADR·승인.
- **닫힌 맵 노드의 캘린더 내 편집** — v1은 열린 맵 노드만 드래그·생성. 닫힌 맵은 읽기 전용 유지.
- **외부 캘린더(Google/CalDAV)·EventKit 전환** — Non-goal(서버 없음·로컬 우선). 하지 않음.
- **월간 성능 캐시** — calendar-view v1의 전체 스캔 한계 승계(느리면 후속).

## 5. 가드레일 정합성

- **"우리는 todo 앱이 아니다"** — 새 라벨/우선순위 등급/프로젝트 없음. 기존 노드 필드만 쓴다.
- **G3 리트머스** — 캘린더·집중을 꺼도 맵·노트·할 일은 온전. 캘린더는 노드의 시간 렌즈일 뿐 core를 종속시키지
  않는다. (core인 `scheduleAt`이 집중을 종속시키는 방향은 0011에서 허용됨.)
- **색 억제** — 새 색 없음. 기존 시맨틱 칩(일정=주황)·scheduleInfo 상태·집중 게이지 재사용.
- **reminder 불변조건** — 모든 경로가 `mapStore` 경유.

## 6. 완료 기준

- [ ] 월간·주간에서 항목을 다른 날짜로 드래그하면 `scheduleAt`이 바뀌고, `reminderOn`이면 리마인더 due도 함께 이동한다.
- [ ] 주간 뷰가 항목을 시간대 위치에 배치한다(종일은 상단 띠, 시각지정은 세로 위치).
- [ ] 주·월 뷰의 항목에서도 집중 시작·완료가 된다.
- [ ] 활성 맵 탭이 있을 때 빈 셀 클릭으로 그 맵에 일정 노드를 만들 수 있고, 없으면 안내가 뜬다.
- [ ] 각 날짜 셀과 일간 헤더에 그 기간의 실제 집중 시간이 계획과 함께 1초 안에 읽히게 표시된다.
- [ ] overdue이면서 집중 0인 항목이 시각적으로 구분되고, 드래그로 오늘·미래로 재계획된다.
- [ ] `MindMapDoc.version` 변경 없음 — 순수 계산 + 기존 필드만 사용(diff로 확인).
- [ ] 드래그 날짜 계산과 날짜↔집중시간 매핑을 순수 함수로 분리해 `calendarMath`/새 모듈에 단위 테스트.
- [ ] `e2e/calendar-*.spec.ts`에 (a) 드래그 리스케줄 → scheduleAt 변경, (b) 계획vs실행 표시 시나리오 추가.

## 7. 열린 질문 (구현 계획 단계에서 확정)

- 빈 셀 생성 시 활성 맵이 여러 개면 어느 탭에 넣나 — v1은 "현재 활성 탭". 별도 "인박스 맵" 개념은 후속 검토.
- 주간 시간 그리드의 시간 범위 — 0–24h 전체 vs 업무시간대 압축 + 접기. 정보 밀도 원칙상 후자 유력.
- 드래그 중 리마인더 due 이동을 낙관적 반영할지, sync 라운드트립 후 반영할지(UX 지연 vs 정합성).
