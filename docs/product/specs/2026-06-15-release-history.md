# 배포 내역 — 기록 + 사용자 가시성

> 작성: 2026-06-15 · 상태: 명세(합의 완료, 구현 예정) · 출처: 사용자 — "배포 히스토리를 기록하고 사용자도 다 알게".

## 1. 문제

배포 기록이 **GitHub Releases**(공개지만 사용자가 잘 안 감)와 `docs/release/notes/`(내부 문서)에 흩어져 있고, **앱 안에서 사용자가 볼 길이 없다.** 자동 업데이트로 새 버전을 받아도 *무엇이 바뀌었는지* 모른다.

## 2. 해결 — 한 번 쓰고 세 곳에서 재사용

### 2.1 기록(Single source) — `CHANGELOG.md` (저장소 루트)
표준 [Keep a Changelog] 형식으로 모든 버전을 한 파일에 누적:
```
## [0.5.2] - 2026-06-15
### 버그 수정
- …
```
릴리즈 본문(GitHub) · `docs/release/notes/` 와 같은 내용. 릴리즈 프로세스에 "CHANGELOG 갱신" 한 줄 추가.

### 2.2 데이터 출처 — 앱에 번들
빌드 시 `CHANGELOG.md`를 렌더러에 **번들**(`?raw` import)하고 순수 함수로 파싱(`[{version, date, body}]`). → **네트워크/GitHub API 불필요**, 오프라인 동작. 자동 업데이트하면 새 빌드의 갱신된 CHANGELOG가 함께 온다. *현재 버전 = CHANGELOG 최상단 항목.*

### 2.3 사용자 가시성 — 두 지점
- **업데이트 직후 "새로운 점" 카드** *(핵심)*: 앱 시작 시 `localStorage.lastSeenVersion` 과 최상단 버전을 비교. 더 높으면(=방금 업데이트됨) 그 버전의 변경점을 카드로 **1회** 보여주고 `lastSeenVersion` 갱신. *최초 설치(저장값 없음)는 카드 없이 버전만 기록.*
- **"업데이트 내역" 오버레이**: 설정 패널 하단 `버전 X.Y.Z · 업데이트 내역` → 전체 버전 목록(버전·날짜·변경점, 기존 `renderMarkdown` 재사용).

## 3. 범위에서 뺀 것

- **GitHub API 실시간 조회** — 설치 버전까지의 기록이면 충분(그보다 최신은 업데이트해야 봄). 번들이 더 단순·견고.
- 다국어 릴리즈 노트, 알림/뱃지, 변경점 자동 생성(노트는 사람이 쓴다).

## 4. 완료 기준

- `CHANGELOG.md` 한 파일이 전 버전 기록(0.2.0–현재).
- 업데이트 후 첫 실행에서 "새로운 점" 카드 1회(최초 설치엔 없음, 닫으면 다시 안 뜸).
- 설정 → "업데이트 내역"으로 전체 버전·변경점 열람.
- 네트워크 없이 동작.
- 파싱은 순수 함수 + 단위 테스트. 일반 사용 회귀 없음.
- 릴리즈 프로세스(RELEASE-PROCESS.md)에 "CHANGELOG 갱신" 단계 추가.
