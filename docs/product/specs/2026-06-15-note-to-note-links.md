# 노트 ↔ 노트 연결 — 기능 명세 (기획)

## 1. 문제 (Why)

지금 연결은 **노드 → 노트** 한 방향뿐이다(노트 frontmatter의 `links: NoteLink[]`, 노드 칩으로 노출). 노트끼리는 엮을 수 없어, 생각이 흩어진 메모를 networked-note 앱처럼 잇지 못한다. 양방향 노트 링크는 경쟁사 리서치에서 꼽은 핵심 차별점이며, "로컬 파일·읽기 쉬운 마크다운" 원칙(ADR 0003)과도 양립한다.

## 2. 해결 (What)

본문에 **인라인 `[[제목]]` 멘션**을 1차 메커니즘으로 둔다. 생각이 적히는 자리에서 바로 다른 노트를 부른다.

- **작성**: 본문에서 `[[` 입력 → 캐럿 위치에 노트 선택 피커(기존 `NoteLinkPicker`의 노트 목록·퍼지검색 재사용). Enter로 삽입. 슬래시 메뉴 "노트 링크"도 제공.
  - 피커는 **현재 노트·집중/work-log 노트 제외**(사이드바 숨김 규칙과 동일).
  - 일치 없으면 "새 노트 '제목' 만들어 연결"(노드→노트 `createAndLink`와 동일 동작, 같은 폴더 규칙 ADR 0009 따름).
- **렌더(에디터·미리보기 공통)**: `[[제목]]` → 클릭 가능한 인라인 칩(`--state-note` 색, 탭/사이드바 노트 정체성과 일관). 대상이 삭제됐으면 흐린/취소선 칩 + "없는 노트" 툴팁(텍스트는 보존). **이동 동작은 §2.5(연결성)를 따른다** — 기본은 탭 교체가 아니라 "옆에 열기".
- **백링크(상단 배치)**: 노트 상단 **"정보" 패널**(연동 옆 정보 버튼 → 본문 위 아코디언)의 한 섹션으로 "이 노트를 참조한 노트"를 보여준다. 인덱스에서 역참조 조회 → 클릭은 §2.5를 따른다(옆에 열기). **여기가 핵심 보상** — 한쪽만 링크해도 양쪽에서 오간다. (배치·스타일은 [[note-meta-header]] 기획을 따른다 — 목차·노드 연결과 한 패널에서 일관되게.)

### 2.5 연결성 — 맥락을 끊지 않는 이동 (사용자 핵심 요구)

문제: 링크를 누르면 현재 탭이 **다른 노트로 교체**되어, 보고 있던 노트와의 연결이 시각적으로 끊긴다("연결했는데 오히려 끊긴 느낌"). 해결은 **원본을 화면에 남기는 것**.

- **기본 동작 = 원본의 "반대쪽 패널"에 열기**: 링크/백링크 클릭 → 대상 노트를 **원본이 있는 패널의 반대쪽**에 연다. 원본이 왼쪽이면 오른쪽, 원본이 오른쪽이면 **왼쪽**(이게 핵심 — 무조건 오른쪽에 여는 `openInRight`는 원본이 오른쪽일 때 원본을 덮어 맥락이 끊긴다). 분할 전이면 분할을 만들어 반대쪽에. → **원본은 항상 화면에 남는다.**
  - **대상 쪽에 이미 다른 노트가 있으면**: 대상을 그 그룹의 **새 활성 탭으로 추가**(스택), 기존 노트는 **배경 탭으로 보존**(덮어쓰기 아님, 탭 클릭으로 복귀). 대상이 곧 그 노트면 중복 없이 활성화만.
  - 구현: `openInRight`를 일반화한 `openBeside(path, sourceGroup)` — 대상 그룹 = `sourceGroup === 0 ? 1 : 0`. 나머지(탭 추가·활성·분할 생성)는 기존 로직 재사용.
  - 수식어 키: **⌘-클릭 = 같은 탭으로 완전 전환**(맥락 교체를 명시적으로 원할 때), 일반 클릭 = 반대쪽에 열기.
- **호버 미리보기(글랜스)**: 링크 칩에 마우스를 올리면 **작은 미리보기 팝오버**(제목 + 첫 단락 발췌)가 떠 이동 없이 내용을 확인. 떠나면 사라짐. 읽기 전용 렌더(markdown.tsx) 재사용.
- **되돌아오기**: 옆에 열기라 "뒤로"가 거의 불필요하지만, 분할 대상 탭에는 닫기(×)로 즉시 원복. (히스토리 기반 뒤로가기는 후속.)
- **차별점**: 이 "나란히 + 글랜스"가 단순 점프(노션·옵시디언 기본)보다 연결성을 강하게 만든다. 경쟁사 리서치의 차별 포인트와 직결.

## 3. 데이터 모델 / 저장 (How)

- **본문이 진실의 원천**, frontmatter는 비정규화 색인. (집중 노트의 🎯 본문 줄 ↔ frontmatter 구조 사본과 같은 패턴.)
- 저장 시 본문의 `[[ ]]`에서 대상 **note id 집합을 추출**해 frontmatter `noteRefs: string[]`에 미러링 → 인덱스는 지금처럼 **frontmatter만** 읽어 백링크가 싸다.
- 조인 키는 **안정적인 note id**(frontmatter). 제목은 표시·마크다운 토큰일 뿐.
- 추가: `NoteDoc.noteRefs?: string[]`, `NoteMeta.noteRefs`, `workspaceStore.noteById(id)` / `backlinks(noteId)` 셀렉터.

### 링크 해석 방식 — **안 A 확정** (2026-06-16, 구현 착수 시 ADR 0010)

본문 표기 `[[제목]]`, 인덱스에서 제목→id 해석. **이름 변경 시 참조 노트들의 `[[옛제목]]`을 일괄 치환**(인덱스+리네임 파이프라인 보유)해 견고성을 확보한다. ADR 0003의 "읽기 쉬운 마크다운" 원칙에 가장 맞고 Obsidian과 동일해 이식성이 높다. 제목 충돌은 드물며 접미사로 구분. (대안: B = `[제목](./파일.md)` 경로 기반, C = `[제목](mindmap://note/<id>)` id 내장 — 일괄 치환이 과하게 까다로울 때의 후퇴안.)

## 4. 범위에서 뺀 것 (후속)

- 그래프 뷰, 링크 호버 미리보기 팝오버, 부분 입력 자동완성 고도화, 제목 충돌 해소 UI.
- 노드 ↔ 노트 기존 링크 모델은 **그대로** 둔다(이 기능은 노트↔노트 전용).

## 5. 완료 기준 (v1)

- [x] 본문 `[[` → 피커 → `[[제목]]` 삽입. 매칭 없으면 **"새 노트 '제목'" 행 → 같은 폴더에 생성·링크**(ADR 0009). 미해결 링크 클릭 시에도 생성→peek.
- [x] 칩 클릭 → **peek**(기본, 이동 없음) → peek의 "열기" → **원본 반대쪽 패널**에 열림(원본 항상 유지), 그쪽 기존 노트는 배경 탭으로 보존
- [x] **⌘/Ctrl-클릭 = peek 건너뛰고 바로 반대쪽 패널에 열기**(본문 위키링크 + 백링크 칩 양쪽)
- [x] **없는/삭제된 대상 = 흐린(점선) 칩** — 대상 생성·삭제 시 인덱스 변경에 맞춰 즉시 갱신(에디터 + 읽기전용 미리보기). 흐린 칩 클릭은 생성 흐름으로.
- [x] plain-text 저장 → 마크다운 왕복 무비용(데코레이션 방식, 커스텀 노드 없음)
- [~] 링크 칩 호버 → 미리보기 *(2026-06-16 **넣지 않기로 결정** — 클릭=peek의 차분함 유지, 마우스 지나갈 때 peek 자동 노출이 결에 안 맞음)*
- [x] 백링크 인덱스 — `NoteMeta.refs`(본문 `[[ ]]` 대상, index 빌드·매 저장 시 갱신), `workspace.backlinks()` 셀렉터. *(frontmatter 영구 미러링 대신 in-memory refs로 충분)*
- [x] 노트 **상단 "정보" 패널**(연동 옆 버튼 → 아코디언, count 배지)에서 백링크 + 연결된 노드 보이고, 백링크 클릭 → peek. 백링크 없으면 버튼 숨김.
- [x] 저장 시 tiptap-markdown의 `\[\[ \]\]` 이스케이프를 언이스케이프 → 디스크에 깨끗한 `[[ ]]`(모든 위키링크 기능의 전제)

## 6. 리스크

- **tiptap ↔ 마크다운 왕복**: `[[ ]]`용 커스텀 인라인 노드의 직렬화/파싱 필요(tiptap-markdown 확장).
- **이름 변경 일괄 치환의 경합**: 열린 탭 버퍼도 함께 갱신해야 함(title→filename 동기화 인프라 재사용).
- **본문 파싱 비용**: 추출은 저장 시 1회뿐 → 인덱스는 frontmatter만 읽어 영향 없음.
