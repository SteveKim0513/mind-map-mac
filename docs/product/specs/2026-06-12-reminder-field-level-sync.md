# 리마인더 동기화 — 필드 단위 충돌 해소 — 기능 명세

> 작성: 2026-06-12 · 상태: 구현·검증 완료
> 출처: 사용자 보고(앱에서 완료한 것이 미리알림에 반영 안 됨) + 사후 검토에서 발견한 데이터 손실급 결함.

## 1. 문제 (Why)

기존 reconcile는 노드와 미리알림을 **레코드 통째로** 비교해 "pull 아니면 push" 한쪽만 골랐다(최근 수정 시각 tiebreak). 그 결과 두 동기화 사이에 **양쪽에서 서로 다른 필드를 바꾸면 한쪽 편집이 통째로 사라진다** — 예: 앱에서 완료(done) 처리하고 같은 사이에 미리알림에서 제목을 바꾸면, recency tiebreak에서 진 쪽의 변경(완료 또는 제목)이 폐기된다. 이것이 "앱에서 완료한 게 미리알림에 반영 안 됨"으로 나타날 수 있는 경로다.

(참고: base가 정상일 때의 단순 outbound 완료는 기존 코드에서도 동작함을 격리 환경에서 확인했다. 이 수정은 그 경로를 견고히 하고, 동시 편집 손실이라는 진짜 결함을 없앤다.)

## 2. 해결 (What)

[`resolveReminder`](../../src/sync/resolveReminder.ts) 순수 함수로 **필드(title/due/done)를 독립적으로** 해소한다.

- 각 필드마다: 로컬이 base 대비 바뀌었나 / 원격이 바뀌었나를 따져, 한쪽만 바뀌었으면 그 값을 채택, **둘 다 바뀐 같은 필드만** 최근 수정 시각으로 tiebreak.
- 결과를 병합해서 needPush(미리알림에 보낼 게 있음)·needPull(노드에 받을 게 있음)을 각각 적용 → 비충돌 동시 편집은 양쪽 모두 보존.
- 편집 중인 노드의 제목은 절대 덮어쓰지 않음(기존 가드 유지).
- 로직을 reconcile에서 분리해 **단위 테스트 7건**으로 회귀 보호(이전엔 0건).

## 3. 범위에서 뺀 것

- 미리알림 생성 시 제목이 아직 비어 빈 제목으로 만들어지는 타이밍(다음 폴에서 자동 교정됨 — 별도 개선 후보).
- 동기화 동시성/큐 구조 변경.

## 4. 완료 기준

- [x] 미리알림 완료해제 → 앱 반영 (실기기 30초 내 확인)
- [x] 앱 done + 미리알림 title 동시 변경 시 양쪽 보존 (단위 테스트)
