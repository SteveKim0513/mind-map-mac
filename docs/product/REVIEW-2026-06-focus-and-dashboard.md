# 서비스 점검·진단 + UX 개선안 — 집중 세션 / 작업 기록 대시보드

> 작성: 2026-06-12 · 작성자: 기획(Claude) · 상태: 진단 보고 (구현 전, 합의 후 선별 반영)
> 방법: 제품 정의·운영 계획·기능 인벤토리 대조 + 실제 컴포넌트 코드 정독(`src/focus/*`). 기존 3계층 토큰 디자인 시스템·다크 모드는 **건드리지 않는 전제**.
> 범위: 새로 추가된 표면 — **집중 세션 플로우**(시작→타이머 pill→세션 노트→종료→완료 카드)와 **작업 기록 대시보드**(`WorkHistory`). 핵심 여정 1~3(맵 작성·리마인더·노트 연결)은 [UX-AUDIT-2026-06](UX-AUDIT-2026-06.md)에서 이미 감사 완료.

## 0. 한 줄 진단

제품의 정체성(로컬·키보드·생각/기록/할일 통합)과 핵심 여정은 탄탄하고 이미 정비되어 있다. 지금의 진짜 쟁점은 *기능 품질*이 아니라 **집중 세션이 추가한 "실행/시간" 축이 제품의 감정적 계약을 바꾸고 있다는 점**, 그리고 **그 새 표면이 아직 여정 감사를 거치지 않았다는 점**이다.

## 1. 평가 (진단)

### 1.1 가장 중요한 전략적 긴장 — "내 사고 도구" → "평가받는 기록"

[PRODUCT-DEFINITION](PRODUCT-DEFINITION.md)은 이 앱을 *개인 지식 작업자의 사적인 사고 도구*(소유·속도·통합)로 규정한다. 반면 [작업 기록 대시보드 재기획](specs/2026-06-12-work-dashboard-redesign.md)은 1차 사용자를 "본인"으로 두되, **결정적 맥락을 "관리자에게 보여주고 평가받는다"**로 확정했다.

이건 드리프트가 아니라 의도된 결정이다. 그러나 **감정적 계약을 바꾼다**:

- "내가 자유롭게 생각하는 공간"과 "내가 측정·평가당하는 기록"은 **다른 제품**이다.
- 후자는 사용자가 무의식적으로 *기록이 좋아 보이도록* 행동하게 만들고, 측정되지 않는 사고(브레인스토밍·탐색·막다른 길)를 위축시킬 위험이 있다.
- 개별 후속 결정(유휴 자동 추적 → 자동 점수화 → 알림 압박)은 각자 합리적이라, 사후에 막을 명분이 약하다.

→ **대응: 가드레일을 명문화한다(§2).** 이 긴장은 "없애야 할 문제"가 아니라 "관리해야 할 설계 상수"다.

### 1.2 기능 표면 응집도 — 양호

- "집중" 단어 충돌(보기 필터 vs 세션)을 라벨 분리로 해소: **"이 노드만 보기"**(보기 필터) / **"집중 세션 시작"**(시간 추적). [decisions/0006](../decisions/0006-focus-session-and-rename.md).
- 진입점 매트릭스([decisions/0004](../decisions/0004-entrypoint-matrix.md))를 새 기능에도 적용(우클릭 메뉴 = 카탈로그, ⌘K, 노드 칩).
- `work-log/`를 사이드바에서 숨기고 **불변**(이름·이동·연동 변경 불가)으로 둔 것은 "신뢰할 수 있는 기록" 목적과 정합적.

이 영역은 기획적으로 깔끔하다. 추가 작업 없음.

### 1.3 가장 큰 미충족 — 콜드 스타트와 "1급 산출물"

- **콜드 스타트:** 대시보드 효용은 세션이 쌓여야 생긴다. 신규 사용자는 데이터 0이라 가치를 못 본다. 사이드바의 "작업 기록" 항목 자체가 첫 사용자에겐 *빈 방으로 가는 문*이다.
- **1급 산출물 미달:** 재기획이 *1급 기능*으로 못 박은 "주간 요약 내보내기"가 현재는 마크다운을 노트에 덤프하는 수준(`buildReportMarkdown`)이라, "관리자에게 한 장으로 내놓는 산출물"이라는 약속에 못 미친다(인사이트·우선순위 요약 누락).

### 1.4 측정 부재

성공 기준(주간 활성·리마인더 동기화율·노트 연결율·집중 기능 사용율)은 정의됐으나 계측이 없다. dogfooding 단계라 수동 관찰로 충분하지만, "차별 기능이 실제로 쓰이는지"를 보려면 최소한의 로컬 이벤트 카운트라도 둘 가치가 있다. (지금은 우선순위 낮음, 인지만.)

### 6.1 E2E 검증 (격리 환경, 빌드 앱 실구동)

`MINDMAP_USER_DATA`/`MINDMAP_REMINDER_LIST` 격리 환경에서 빌드 앱을 Playwright로 구동, 세션 시작→노트→종료→완료 카드→대시보드 라운드트립을 직접 관찰.

- **통과**: B2(첫 세션 코치마크 토스트), B7(완료 카드 "최근 7일 · N일 집중" 중립 표기), B3(회고 입력 후 **Esc로도 저장**됨 — frontmatter `reflect` 확인), B5(시간대 축 `0·6·12·18·23` 렌더), B4(내보내기 노트에 `## 한눈에` 인사이트 + `## 마감 vs 실제 집중` 표 포함), B8(데이터 0 빈 상태), 대시보드 일지·주제별·마감대조·질·추세 정상.

### 6.2 검증 중 발견·수정한 버그 — 🎯 목표 자리표시자 누출

- **증상**: 완료 카드와 frontmatter `goal`에 사용자가 입력하지 않은 **자리표시자**("무엇에 집중하나 — 한 문장으로*")가 목표로 박힘.
- **원인**: 노트 템플릿은 자리표시자를 `_..._`(밑줄)로 쓰지만 **tiptap가 첫 자동저장에서 `*..*`(별표)로 재직렬화**한다. `extractGoal`은 `_`로 시작하는 줄만 건너뛰어, `*..*` 자리표시자를 진짜 목표로 반환(심지어 그 아래 실제 목표를 가림).
- **수정(최종)**: `extractGoal`을 순수 모듈 `src/focus/goal.ts`로 분리. **단일 진실 공급원** `src/focus/sessionNote.ts`(`SECTIONS`/`BODY_TEMPLATE`/`PLACEHOLDER_HINTS`)를 만들어 템플릿과 추출기가 같은 힌트 문자열을 공유 → 마커 불일치(`_`↔`*`)가 구조적으로 불가능. 추출기는 **마커 제거 후 텍스트가 알려진 힌트와 정확히 일치할 때만** 건너뛴다.
- **추가로 잡은 2차 이슈**: 처음 시도한 "이탤릭이면 무조건 건너뛰기" 방식은, 사용자가 자리표시자를 **지우고 그 위에 목표를 타이핑**하면 목표가 이탤릭(`*목표*`)이 되어 **진짜 목표까지 버려지는** 새 결함을 낳았다. 정확 일치 방식으로 바꿔 해결.
