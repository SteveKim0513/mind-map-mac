## 0. 왜 지금 다시 다루는가

REDESIGN-VISION 이후에도 사용자가 "인식이 안 된다"고 느낀다는 건, 그 비전이 틀려서가 아니라 **다른 층위의 문제를 건드리지 않았기 때문**이다.

- REDESIGN-VISION은 "**입력 과정에서 결정을 없애는 것**"(예: 날짜를 자동으로 인식)에 집중했다 — 이미 도구에 익숙한 사람의 **효율**을 높이는 방향.
- 이번 문제는 "**화면에 있는 것의 정체를 못 알아본다**"는 것 — 도구가 처음이거나 익숙하지 않은 사람의 **인식(recognition)**의 문제다.
- 두 층위는 다르다: 결정을 없애도, 남은 화면 요소가 "이게 뭔지, 눌러도 되는지, 이 버튼과 저 버튼이 왜 다른지"를 스스로 설명하지 못하면 여전히 헤맨다.

이 문서는 아래 순서로 간다: **(1) "UX가 극대화된 UI"를 검증 가능한 기준으로 정의 → (2) 그 기준으로 현재 상태를 코드 근거와 함께 진단 → (3) 기준을 충족시키는 UI 전략을 제안.**

## 1. "UX가 극대화된 UI"란 무엇인가 — 정의

막연히 "직관적으로 만들자"는 진단 기준이 될 수 없다. 아래 7가지를, 현재 UI 요소 하나하나에 참/거짓으로 물을 수 있는 **검증 가능한 조건**으로 정의한다. 이 문서의 모든 진단과 전략은 이 7가지 중 몇 번을 위반/충족하는지로 되짚을 수 있어야 한다.

| # | 조건 | 한 줄 정의 | 반증 가능한 질문 |
|---|---|---|---|
| **U1. 재인 우선 (Recognition over recall)** | 사용자가 기억하지 않아도, 화면을 보는 순간 "이건 누를 수 있고 대략 이런 일을 한다"는 게 보여야 한다 | 이 요소가 클릭 가능하다는 걸 호버하기 **전에** 알 수 있는가? |
| **U2. 기능당 하나의 형태 (One shape per feature)** | 같은 기능은 어디서 진입하든 같은 시각 언어(모양·위치·닫는 법)로 나타나야 한다 | 이 기능을 여는 진입점이 여러 개라면, 그 결과 UI는 매번 같게 생겼는가? |
| **U3. 숨김이 아닌 계단식 공개 (Progressive disclosure, not hiding)** | 기본 화면엔 핵심만, 고급은 "한 단계 더" 로 도달 가능해야 한다 — 단, 1차 행동(primary action)이 툴팁 뒤에만 존재해서는 안 된다 | 툴팁을 절대 못 보는 사용자도 이 기능의 존재를 알 수 있는가? |
| **U4. 즉각적이고 읽히는 피드백** | 모든 조작은 1초 안에 색·아이콘·위치 변화로 결과를 보여줘야 한다 | 클릭 직후, 뭔가 바뀌었다는 걸 설명 없이 알 수 있는가? |
| **U5. 관대함과 되돌림 (Forgiving & reversible)** | 실수의 비용이 낮아야 한다 — 파괴적 동작에만 명확한 경고, 나머지는 즉시 되돌릴 수 있어야 한다 | 잘못 눌렀을 때 Cmd+Z 또는 즉시 취소로 되돌아가는가? |
| **U6. 하나의 어휘 (One vocabulary)** | 같은 개념은 앱 전체에서 같은 이름·같은 아이콘을 쓴다. 내부 구현 용어가 화면에 새어나오지 않는다 | 이 라벨이 내부 코드/데이터 모델 용어가 아니라 사용자가 실제로 쓰는 말인가? |
| **U7. 구조 탈출로 (Rescue path)** | 마우스로 우연히 발견하지 못해도, 검색 가능한 하나의 통로(예: 커맨드 팔레트)로 모든 핵심 동작에 도달할 수 있어야 한다 | 이 아이콘을 평생 못 찾은 사람도, "이걸 하고 싶다"고 검색해서 실행할 수 있는가? |

**핵심 통찰**: U1~U6은 "화면에 보이는 것"의 문제이고, U7은 "화면에 안 보여도 괜찮은" 안전망이다. 지금 앱은 U7이 거의 없어서(§3-4 참조) U1~U6의 결함이 그대로 사용자에게 노출된다 — 즉 두 겹의 방어선 중 하나가 통째로 비어 있다.

## 2. 현재 상태 진단 — 마인드맵 노드 상호작용

(근거: `src/canvas/NodeView.tsx`, `Canvas.tsx`, `SelectionToolbar.tsx`, `src/menu/ContextMenu.tsx`, `src/inspector/*.tsx`, `src/note/NotePopup.tsx` 코드 조사)

### 2-1. 노드에서 나오는 플로팅 UI가 "5개의 서로 다른 가족"이다 (U2 위반)

| UI | 열리는 경로 (진입점 개수) | 시각 형태 |
|---|---|---|
| `ContextMenu` | 우클릭 (1개) | 텍스트 목록 메뉴 |
| `SchedulePopover` | 게이지 칩 클릭 / 툴바 달력 아이콘 / 컨텍스트 메뉴 — **3개** | 카드형 패널(고정폭 300px, 미리보기·토글 포함) |
| `NodePopover`(아이콘 선택) | 컨텍스트 메뉴 "아이콘…"만 — 1개, 노드 위엔 진입점 자체가 없음 | 작은 그리드 |
| `LinkAddPopover` | 툴바 링크 아이콘 / 컨텍스트 메뉴 — 2개 | 한 줄 입력 필드 |
| `NotePopup` | 노트 링크 칩 클릭 | Quick-Look풍 앵커 팝업 또는 중앙 모달(앵커 없을 때) — **한 컴포넌트 안에 2가지 다른 형태** |
| 색상 선택 | 컨텍스트 메뉴 인라인 스와치 행 **또는** 툴바의 별도 스와치 드롭다운 — **같은 기능의 서로 다른 두 구현** | 완전히 다른 두 패턴 |

같은 "일정을 정한다"는 기능이 칩(색이 있는 알약), 아이콘(툴바의 무라벨 달력 기호), 텍스트("날짜·시간 설정…")로 **매번 다른 모습**으로 나타난다. 사용자가 "이게 그 기능이구나"를 학습할 기준점이 없다 — 이것이 "기능간 경계가 모호하다"는 호소의 직접적 원인이다.

추가로 `SchedulePopover`는 z-index 50, `ContextMenu`/`NotePopup`은 z-index 70으로 서로 다르다 — 겹치는 상황에서 어떤 게 위로 뜨는지 일관성이 없다.

### 2-2. 1차 행동이 툴팁 뒤에 숨어 있다 (U1·U3 위반)

- `SelectionToolbar`: 노드를 선택하면 자동으로 뜨는 9개 아이콘 버튼(색상·완료·메모·링크·노트연결·일정·집중·자식추가·삭제) — **전부 라벨 없이 아이콘+네이티브 `title` 툴팁만**. REDESIGN-VISION §T2가 "자주 쓰는 3~4개만 툴바에, 나머지는 메뉴로"라고 이미 지정했지만 실제로는 9개가 그대로 있다 — **설계 의도와 구현이 이미 어긋나 있다.**
- 접기/펼치기 화살표, 링크 제거(×) 버튼: 평소엔 `opacity: 0`으로 **완전히 안 보이다가** 호버해야 나타난다. 존재 자체를 모르면 절대 발견 못 한다.
- 노드 호버 시 커서가 `grab`(드래그 신호)인데, 그 노드는 동시에 클릭·더블클릭 대상이기도 하다 — 커서가 주는 신호와 실제 상호작용 종류가 어긋난다.

### 2-3. 마우스로만 갈 수 있고, 돌아올 방법(rescue path)이 없다 (U7 위반)

일정 설정, 색상 지정, 링크 추가/제거, 노트 연결, 아이콘 지정, 메모, 집중 세션 시작 — 이 중 키보드 단축키나 커맨드 팔레트 항목이 있는 것은 **하나도 없다**(완료 토글 ⌘Enter만 예외). `CommandPalette`의 "노드" 그룹은 검색해서 **선택·이동**만 시킬 뿐 **실행**은 못 한다 — REDESIGN-VISION §3-7에서 "⌘K 안에서 액션 실행"은 명시적으로 범위 제외됐다. 즉 지금은 위 기능들을 작은 아이콘으로 우연히 발견하는 것 외엔 방법이 없다.

## 3. 현재 상태 진단 — 화면별 상세

노드 하나에 국한된 문제가 아니다 — 같은 패턴(같은 기능이 진입점마다 다른 모습, 아이콘 하나로 존재를 감추기, 되돌릴 수 없는 동작에 안전장치 없음)이 설정·우클릭 메뉴·사이드바·노트 에디터·검색·포커스 모드·탭까지 앱 전체에 반복된다. 화면별로 코드 근거와 함께 진단한다.

> **⚠ 가장 시급한 항목 (UX를 넘어 데이터 안전 문제)**: `src/store/sessionStore.ts`의 `closeTab`/`closeAllTabs`는 저장 중(자동저장 1000ms 디바운스 창 안)인지 확인하지 않고 탭을 닫는다. 편집 중 탭을 닫거나 "모두 닫기"를 누르면 마지막 입력이 경고 없이 사라질 수 있다 — 이건 "직관성"이 아니라 **데이터 손실 버그**에 가까우므로, 이 비전 문서의 착수 순서와 무관하게 별도로 우선 수정을 검토할 것을 권한다.

### 3-1. 설정 화면 (`src/ui/Settings.tsx`)

- **U6 위반 — "템플릿"이 한 화면 안에서 두 가지를 가리킴**: "노트 템플릿"(사이드바에 템플릿 폴더를 보여주는 토글, `:248`)과 "정보 양식"(`MetaTemplate`, 사용자 정의 메타데이터 필드, `:288-292`)이 같은 화면에 있는데 코드 전반의 이름(`Template`, `TemplateSummary`, `MetaTemplate`, `TemplateAddButton`/`MetaAddButton`)이 한 어휘 계열이라 화면·에디터 툴바까지 이어지며 혼동을 만든다 (3-4절 노트 에디터 진단과 동일 뿌리).
- **U2 위반 — 그룹핑이 목적이 아니라 내부 시스템 기준**: "고급 설정" 안이 "AI 기능", "정보 양식" 같은 내부 주체 이름으로 나뉘어 있다. 사용자가 하려는 일("AI로 요약하기", "노트에 항목 추가하기") 기준이 아니다.
- **U1 위반 — 라벨이 아예 없는 컨트롤**: AI 제공자 선택 버튼(`:386-391, 441-446`)은 제공자가 하나만 설정된 경우 `title`조차 비어 있다 — 호버해도 아무 설명이 없다. 뒤로가기·닫기 버튼도 아이콘뿐(`:156-163`).
- **U3 애매 — "고급 설정"이 내용 미리보기 없이 숨김**: 펼치기 전엔 그 안에 뭐가 있는지 전혀 알 수 없다(`:267-274`) — 계단식 공개라기보다 그냥 숨김에 가깝다.

### 3-2. 보조 메뉴 전체 (우클릭 등) — 앱 안에 3개의 서로 다른 구현

| 위치 | 구현 | 공유 컴포넌트? | Escape로 닫힘? | 화면 밖 클램핑? |
|---|---|---|---|---|
| 캔버스 노드 우클릭 | `src/menu/ContextMenu.tsx` (정식 컴포넌트) | — | ✓ | ✓ (`useLayoutEffect`) |
| 탭 우클릭 (`TabBar.tsx:166-244`) | 별도 구현, CSS 클래스만 `ctx-menu` 공유 | 부분적 | **✗ 없음** | **✗ 없음 — 화면 밖으로 잘릴 수 있음** |
| 사이드바 파일/폴더 행 | 우클릭 메뉴 **자체가 없음** — 호버해야 나오는 아이콘(즐겨찾기·이름변경·삭제, `Sidebar.tsx:509-544`)뿐 | 없음 | 해당 없음 | 해당 없음 |

"항목에 대한 보조 동작"이라는 같은 개념이 세 가지 다른 상호작용 모양으로 나타난다 — 우클릭을 캔버스와 탭에서 배운 사용자가 사이드바에서도 시도하지만 아무 일도 안 일어난다(U1). 탭 메뉴는 Escape가 안 먹고 화면 경계 보정도 없어 정식 메뉴보다 완성도가 떨어진다(U4).

### 3-3. 사이드바

전반적으로 상태가 좋은 편이다 — 스마트 뷰(오늘/즐겨찾기/최근)는 트리와 의도적으로 다른 모양(`sb-nav-item`, 트위스티·파일아이콘 없음)으로 구분되어 있고(U2 충족), 새 노트/새 마인드맵 생성 경로는 어디서 시작해도 동일한 핸들러로 수렴하며(U2 충족), 삭제는 확인창 없이 되돌리기 토스트로 처리된다(U5 충족).

- **U1 위반**: 즐겨찾기·이름변경·삭제(행 액션, `:511-543`), 라이브러리 생성 버튼 4개(`:596-607`), 사이드바 접기(`:561`), 하단 휴지통/템플릿/설정(`:679-700`) 모두 라벨 없이 아이콘+툴팁뿐.
- **U6 위반 — 라벨 표기 불일치**: 하단 템플릿 배지가 "Note Template"이라는 영어 그대로 노출된다(`:690`) — 나머지는 전부 한글인데 여기만 다르다.
- **U6 위반 — 휴지통 라벨이 실제 동작과 다름**: "영구 삭제" 버튼을 눌러도 실제로는 OS 휴지통으로 이동할 뿐이라고 안내 문구 자체가 말한다(`TrashPanel.tsx:57-58, 69`) — "영구"라는 단어가 실제보다 더 확정적으로 들린다.

### 3-4. 노트 에디터 (`EditorToolbar.tsx`, `NoteMetaBlock.tsx`)

- **U1/U2 위반 — 20개 이상 아이콘이 위계 없이 한 줄로**: 서식(H1-3·굵게·기울임)/구조(목록·인용·코드블록)/삽입(링크·이미지·표·구분선)이 구분선(`md-tb-sep`)만으로 나뉘고 그룹 이름이 없다. 약 8개 컨트롤은 텍스트 없이 아이콘+툴팁뿐.
- **U6 위반 — "노트 템플릿"과 "정보 양식"이 툴바에서 나란히, 거의 같은 모양으로**: 두 버튼(`템플릿+`, `양식+`)이 같은 드롭다운 클래스(`meta-add-menu`)·같은 "+ 새 X 만들기" 문구·같은 아이콘 스타일을 공유한다 — 코드 주석 자체가 개발자에게 "이 둘을 헷갈리지 말라"고 경고할 정도로(`NotePane.tsx:214-216`), 만든 사람도 헷갈리는 개념이다.
- **U5 위반 — 템플릿 삭제가 확인·되돌리기 둘 다 없음**: 메타 템플릿 삭제(`Settings.tsx:519`)는 `window.confirm` 없이 즉시 삭제되고, 그 템플릿을 쓰던 메모 블록은 "⚠ 삭제된 템플릿" 카드로 남는다(부분 완화는 있으나 원본 삭제 자체는 되돌릴 수 없음).
- **U7 미흡**: 굵게/기울임 외 나머지 서식·삽입 단축키는 툴팁에도 없어 마우스로만 접근 가능하다.

### 3-5. 검색·커맨드·전역 캡처 — 4개의 서로 다른 "찾기"

| 화면 | 여는 법 | 찾는 대상 |
|---|---|---|
| 커맨드 팔레트 | ⌘K | 명령 항상 + 열린 맵의 노드(입력 시) + 파일명 |
| Quick Open | ⌘P | 워크스페이스 전체 파일명만 |
| 전역 검색 | ⌘⇧F, 사이드바 검색 버튼 | 워크스페이스 전체의 노드 텍스트 + 노트 본문 |
| 캔버스 내 검색 | ⌘F, 팔레트 "노드 검색" | 열린 맵의 노드만 |

- **U2/U6 위반**: 커맨드 팔레트의 노드 검색과 캔버스 내 검색(⌘F)은 찾는 범위가 거의 동일한데 서로 다른 컴포넌트·단축키다 — 어느 걸 써야 하는지 예측할 원칙이 없다. (다행히 각 화면이 서로를 향한 탈출구 버튼은 갖고 있어 U7은 충족 — 문제는 처음 어디로 들어갈지, 즉 입구다.)
- **U1 위반 — 전역 캡처(⌥Space)가 화면 어디에도 없음**: 등록 여부만 설정에 조용히 표시(`Settings.tsx:317-332`, 주석: "v1엔 토글 없이 상태만 보여줌"), 커맨드 팔레트를 검색해야만 존재를 알 수 있다. 첫 실행 사용자는 이 기능이 있는지 전혀 알 길이 없다.
- **U4 위반 — 캡처 후 어디로 갔는지 안 보여줌**: `CaptureWindow.tsx`는 입력 중에도 목적지(어느 맵/파일)를 안 보여주고, Enter 후에도 확인 없이 창만 사라진다(`:31-62`).
- **U7 위반 — 즐겨찾기/최근 수정이 팔레트에 없음**: `openRecent`/`openFavorites`는 사이드바 버튼에만 연결되어 있고 `buildCommands`(App.tsx:489-542) 목록엔 없다 — "커맨드 팔레트가 하나의 정문"이라는 REDESIGN-VISION §A5 목표와 실제 구현 사이에 이미 벌어진 틈이다.

### 3-6. 포커스 모드 · 탭/분할 · 홈 화면

- **U4 위반 — 포커스 오래-지속 신호가 접힌 상태에선 색 하나뿐**: 접힌 포커스 알약은 점 색(호박/빨강)과 경과 시간만 보여준다(`FocusWidget.tsx:44-53`) — 정작 코드 주석은 "텍스트만으로 의미를 주지 않는다"고 하지만, 접힌 상태엔 그 텍스트조차 없어 실질적으로 색-only 신호가 된다.
- **U3 위반 — 탭을 아래로 드래그하면 분할된다는 걸 미리 알 방법이 없음**: 분할 드롭존은 드래그를 이미 시작해야만 나타난다(`App.tsx:401-411`) — 이건 "누르면 펼쳐지는" 계단식 공개가 아니라 우연한 발견에 의존하는 것. (다행히 커맨드 팔레트에 "화면 분할" 항목이 있어 대체 경로는 있음 — U7 충족.)
- **U5 위반 (위 경고 박스와 동일 사안) — 탭 닫기/모두 닫기에 저장 여부 확인이 없음**.
- **양호한 사례**: 홈 화면은 라벨+아이콘이 짝을 이루고("새 마인드맵"/"새 노트") 빈 상태 안내도 명확하다(U1 충족) — 새 탭/닫기/분할 동작이 3가지 경로(×버튼/우클릭/⌘W)를 거쳐도 모두 같은 핸들러로 수렴한다(U6 충족) — 닫기 아이콘(×)과 삭제 아이콘(휴지통)의 의미 구분도 앱 전체에서 일관됨(U6 충족).

### 정리 — 반복되는 근본 원인

위 6개 화면에서 반복되는 것은 딱 두 가지다: **(1) 기능이 자라날 때마다 진입점을 하나씩 덧붙였을 뿐 기존 진입점을 정리하지 않았고, (2) 라벨을 "호버하면 보이는 title"로만 두고 화면에 항상 보이는 텍스트로 승격시키지 않았다.** 4절의 전략은 이 두 원인을 화면 종류에 상관없이 공통으로 해소하는 걸 목표로 한다.

**선례 확인**: `docs/product/reports/UX-AUDIT-2026-06.md`의 항목들은 이미 해소됐고, 별도의 접근성/성능 감사도 모두 통과 상태다. 즉 "도구 초심자가 겪는 인식 문제"는 **이전에 어떤 감사에서도 다루지 않은 새로운 렌즈**다 — 지금 이 문서가 처음이다.

## 4. UI 전략

각 전략은 위 1절의 조건(U1~U7) 중 무엇을 충족시키는지 표기한다. 전략들은 서로 독립적이며, 부분적으로만 채택해도 각자 효과가 있다.

### 전략 A — 노드뿐 아니라 앱 전체의 플로팅 UI를 3개의 "정식 형태"로 통합 *(충족: U1, U2, U4)*

지금의 5개 이상 임시방편 구현을 **정확히 3개의 형태**로만 제한한다 — 그리고 이 통합의 범위는 노드 팝오버에 그치지 않는다. **탭 우클릭 메뉴**(Escape 미지원·화면 밖 클램핑 없음, 3-2절), **사이드바의 없는 우클릭 메뉴**(호버 아이콘으로만 대체되어 있음)도 같은 기반으로 들어와야 "우클릭하면 보조 동작이 나온다"는 학습이 앱 전체에서 배신당하지 않는다.

| 형태 | 쓰는 경우 | 예 |
|---|---|---|
| **목록형(List)** | 여러 동작 중 하나를 고르는 경우 | 캔버스 우클릭 메뉴, 탭 우클릭 메뉴, 사이드바 행 보조 메뉴(신설) |
| **카드형(Card)** | 값을 보고/입력하는 경우 | 일정, 링크 추가, 노트 미리보기 |
| **인라인 스와치(Inline swatch)** | 시각적으로 바로 비교하며 고르는 경우 | 색상 — **하나로 통일, 컨텍스트 메뉴판과 툴바판 중 하나만 남긴다** |

셋 다 같은 위치 계산·같은 열림/닫힘 애니메이션·같은 바깥 클릭/Esc 규칙을 공유하는 하나의 기반(예: `usePopover`/`<Sheet>`) 위에서 동작해야 한다 — REDESIGN-VISION §3-5에서 유예했던 바로 그 통합이며, 이번 문서가 "왜 지금 해야 하는가"의 근거(사용자 체감 혼란 + 탭 메뉴의 실제 결함)를 제공한다.

### 전략 B — "조용하지만 안 보이지는 않는" 기본값 *(충족: U1, U3)*

`opacity: 0` (완전히 안 보임) 대신, 항상 낮은 투명도(예: 30~40%)로 존재를 암시하고 호버 시 100%로 올라가는 방식으로 통일한다. "존재를 모르면 못 찾는" 컨트롤(접기 화살표, 링크 제거)을 "약하게라도 늘 보이는" 컨트롤로 바꾼다. 자식 추가(+) 버튼은 이미 이 패턴(`opacity: 0.55` → 호버 시 1)을 쓰고 있으므로, 그걸 나머지 컨트롤의 기준으로 확산한다.

### 전략 C — 진입점은 하나의 대표 + 하나의 백업만 *(충족: U2, U6)*

같은 기능이 3개의 서로 다른 모습으로 흩어지는 대신, **하나의 대표 진입점**(예: 게이지 칩 — 이미 값이 있을 때만 보임)과 **하나의 백업 진입점**(컨텍스트 메뉴 — 항상 텍스트 라벨로 존재)만 남긴다. 툴바 아이콘과 메뉴 텍스트가 같은 기능을 중복 표현하는 지금 구조를 정리한다. 남기기로 한 아이콘은 그 기능이 활성화됐을 때 나타나는 칩과 **같은 아이콘·같은 색**을 써서, "이 아이콘을 누르면 저 칩이 생기는구나"를 한 번에 학습하게 한다.

### 전략 D — 커맨드 팔레트를 모든 노드 동작의 탈출구로 *(충족: U7)*

지금 "노드"를 검색하면 선택·이동만 되는 걸, 선택된 노드에 한해 "일정 설정", "색상 지정", "링크 추가", "노트 연결", "집중 시작" 같은 실행 가능한 동작으로 확장한다. 이렇게 하면 U1~U3을 완벽히 못 지켜도(아이콘을 못 알아봐도) 항상 "이걸 하고 싶다"고 타이핑해서 도달할 수 있는 안전망이 생긴다 — 초심자와 숙련자 모두에게 같은 통로가 하나 더 생기는 것.

### 전략 E — 실제 조작 기반 첫 실행 가이드 *(충족: U1, U3)*

> **보류 (2026-07-16, [결정 0013](../decisions/0013-onboarding-after-features-stable.md))**: 이 전략을 구현한
> `FirstRunCoach`가 v0.10.0에 잠깐 들어갔다가 제거됐다. 온보딩은 핵심 기능 동선이 **안정된 뒤**에 설계한다
> — 기능이 바뀌는 중엔 안내가 곧 낡기 때문. 재도입 시 이 전략과 결정 0013을 함께 갱신한다.

모달 설명서가 아니라, **처음 빈 워크스페이스를 만들었을 때만** 실제 캔버스 위에서 순서대로 나타나는 스팟라이트형 안내(예: "여기 클릭해서 첫 생각을 적어보세요" → 입력 후 "Tab을 누르면 하위 생각이 생겨요") — 기존의 `focusCoachShown`/`lastSeenVersion` 같은 1회성 로컬 플래그 패턴을 그대로 재사용할 수 있다. 언제든 건너뛸 수 있고, 한 번 끄면 다시 안 뜬다. **"이미 몇 번 써본 사람"에게만 뜨는 지금의 `GrowthNudges`와 달리, 이건 정확히 첫 순간을 겨냥한다.**

### 전략 F — 사용자 언어로 번역 *(충족: U6)*

"메타 템플릿"→ 사용자가 실제로 하는 일 기준으로("정보 양식" 등, 정확한 표현은 별도 카피 감사에서 확정), "메타 블록"→ 마찬가지. 이름이 겹치는 "템플릿"이라는 단어를 두 기능 중 하나에서 다른 이름으로 바꿔 혼동을 없앤다. 이 전략은 코드 변경 없이 라벨 문자열 교체만으로 상당 부분 가능해서, 가장 낮은 리스크로 먼저 시도해볼 수 있다.

### 전략 G — 아이콘·색 1개 = 의미 1개 계약 *(충족: U1, U2, U6)*

지금 색상 선택 UI가 두 벌 존재하듯, 같은 개념에 서로 다른 시각 표현이 붙는 걸 원천 차단하는 규칙을 문서화한다: "이 아이콘/색은 앱 전체에서 이 의미로만 쓴다"는 계약을 `docs/design/COLOR-SYSTEM.md`에 이미 있는 3계층 색 시스템과 나란히, **아이콘 버전으로 확장**한다. 새 UI를 만들 때 새 아이콘을 즉흥적으로 고르지 못하게 막는 게 목적이다.

### 전략 H — 파괴적 동작 전수 계약: "확인 아니면 되돌리기, 둘 다 없는 건 금지" *(충족: U5)*

이번 조사에서 확인된 위반 사례만 셋이다 — **탭 닫기/모두 닫기**(저장 중 여부 확인 없이 즉시 닫힘, 자동저장 디바운스 창 안이면 데이터 손실), **메타 템플릿 삭제**(확인창도 되돌리기도 없음), **휴지통 "영구 삭제" 라벨**(실제로는 OS 휴지통 이동인데 문구가 더 확정적으로 들림). 하나의 규칙으로 통일한다: 되돌릴 수 없는 동작은 반드시 확인을 받거나(모달/네이티브 confirm), 확인 없이 진행한다면 반드시 되돌리기 토스트를 제공해야 한다 — **둘 다 없는 경로는 존재해선 안 된다.** 이미 사이드바 삭제(확인 없이 되돌리기 토스트)와 휴지통 영구삭제(확인창 있음)는 이 규칙을 충족하고 있으므로, 나머지 위반 사례를 그 기준에 맞추면 된다. 라벨 문구도 실제 동작과 정확히 일치시킨다("영구 삭제" → 실제로 앱 안에서 되돌릴 수 없다는 뜻이면 그렇게, OS 휴지통일 뿐이면 "휴지통으로 이동" 등으로).

### 전략 I — 밀집 툴바에 시각적 위계 부여 *(충족: U1, U2)*

`EditorToolbar`처럼 20개 이상의 아이콘이 구분선만으로 나뉜 채 한 줄에 늘어선 곳은, 그룹마다 옅은 배경/여백 차이 또는 짧은 그룹 레이블(예: 마우스 오버 시 그룹 제목이 보이는 것이 아니라 그룹 자체에 은은한 배경을 줘 "서식"과 "삽입"이 시각적으로 다른 구역임을 항상 보이게)을 부여한다. 텍스트가 있는 버튼(H1·B·I 등)과 아이콘만 있는 버튼이 같은 줄에서 시각적 무게가 똑같은 것도 조정 대상 — 자주 쓰는 서식(굵게·기울임·목록)은 항상 보이는 라벨/글자를 유지하고, 덜 쓰는 삽입류(표·구분선)는 그룹 안에 모아 압축한다.

### 전략 J — 설정 화면을 "내가 하려는 일" 기준으로 재편 *(충족: U2, U6)*

"고급 설정" 아래를 "AI 기능"·"정보 양식" 같은 내부 주체 이름 대신, 사용자가 하려는 일 기준 카드로 재편한다(예: "AI로 요약·정리하기", "노트에 반복되는 항목 만들기"). "노트 템플릿"과 "정보 양식"이 같은 화면에서 다른 이름으로 확실히 구분되도록, 전략 F(용어 번역)와 함께 적용한다.

## 6. 성공 기준 (검증 방법)

이 비전이 실제로 작동했는지는 다음으로 확인한다 — 구현 후 `ux-audit` 스킬로 "도구 사용 경험이 없는 초심자" 페르소나를 명시적으로 지정해 재감사한다.

- 처음 앱을 여는 사람이 문서·타인의 설명 없이: (a) 노드를 만들고, (b) 그 노드에 일정을 걸고, (c) 색을 바꾸고, (d) 노트를 연결할 수 있는가.
- 같은 기능을 서로 다른 진입점(칩/툴바/메뉴)으로 열었을 때, 뜨는 UI가 "이거 아까 그거네"라고 알아볼 만큼 같은가.
- 아이콘의 뜻을 몰라도, 커맨드 팔레트에 하고 싶은 말을 치면 실행까지 갈 수 있는가.
- 캔버스·탭·사이드바 어디서 우클릭해도, "보조 동작이 나온다"는 같은 기대가 배신당하지 않는가.
- 앱 안의 되돌릴 수 없는 동작(삭제류)을 모두 나열했을 때, 확인창 또는 되돌리기 토스트 둘 중 하나가 반드시 있는가 — 예외가 하나도 없는가.
- 설정 화면의 "고급 설정"을 열기 전에도, 그 안에 대략 뭐가 있을지 짐작할 수 있는가.

## 7. 비범위 (Non-goals)

- 이 문서 자체는 어떤 코드도 바꾸지 않는다.
- `.mind`/`.md` 파일 포맷, `MindMapDoc.version` 스키마는 대상이 아니다.
- 키보드 우선·macOS 네이티브 정체성(REDESIGN-VISION의 기존 원칙)을 바꾸지 않는다 — 오히려 전략 D·G는 그 정체성을 더 철저히 지키는 방향이다.
- 전략 A·D·E처럼 새 공유 컴포넌트/새 상태가 필요한 항목은, 착수 시점에 `docs/exec-plans/active/`에 별도 실행 계획을 먼저 쓴다(본 문서는 그 전 단계의 방향 합의용).
- 탭 닫기 데이터 손실 건(3절 경고 박스)의 실제 수정은 이 문서의 승인을 기다리지 않는다 — 별도 버그 수정으로 처리 가능하며, 실제 재현 여부 확인 후 `.claude/rules/testing.md`의 버그 수정 계약(재현 테스트 우선)을 그대로 따른다.
