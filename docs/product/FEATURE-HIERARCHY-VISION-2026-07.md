# 기능 계층 재정의 — 정리 · 실행 · 통찰 3축

> 작성: 2026-07-15 · 상태: **초안 — 방향 합의 필요 (구현 전)**
> 근거: [결정 0011](../decisions/0011-three-axis-thinking-model.md), 사용자가 직접 제시한 3축 재정의.
> 관계: [UX-CLARITY-VISION-2026-07.md](UX-CLARITY-VISION-2026-07.md)를 대체하지 않는다 — 그 문서는
> "같은 기능이 어디서 열든 같은 모양이어야 한다"(시각적 일관성)였고, 이 문서는 "기능이 애초에 왜 이
> 그룹에 속하는가"(개념적 계층)다. UX-CLARITY가 만든 공유 컴포넌트(ColorSwatchGrid, useDismissablePosition
> 등)는 그대로 재사용한다 — 이 문서는 그 위에 얹는 재배치 작업.

## 0. 왜 다시 여는가

지난 UX-CLARITY-VISION 작업은 "우클릭 메뉴/선택 툴바/색 선택기가 어디서 열려도 같은 모양"이라는
**시각적 일관성**을 만들었다. 하지만 사용자가 지적한 문제는 그보다 한 겹 아래에 있다 — **"완료·일정·
집중이 왜 서로 다른 그룹에 있는가", "메모·링크·노트연결을 '첨부'로 묶은 게 맞는가"** 같은, 그룹 자체의
존재 이유에 대한 질문이다. 시각적으로 아무리 일관돼도, 그룹핑 기준 자체가 사용자의 실제 사고 흐름과
안 맞으면 "정리가 안 된 느낌"은 사라지지 않는다.

## 1. 세 축 (요약 — 전체 정의는 [결정 0011](../decisions/0011-three-axis-thinking-model.md))

- **정리(Organize)** — 생각을 포착·구조화
- **실행(Execute)** — 정리된 것 중 할 일을 실제로 해냄 (완료→일정→집중이 하나의 흐름)
- **통찰(Insight)** — 더 깊이 쓰고 되짚어 이해

## 2. 전체 기능 재평가 — 축 매핑

`docs/product/FEATURE-INVENTORY.md`의 16개 영역을 축 기준으로 재분류한다. "축 무관"은 세 축
어디에도 속하지 않는 **인프라/내비게이션** — 파일을 열고 검색하고 창을 나누는 것 같은, 사고 자체가
아니라 사고를 담는 그릇에 해당한다.

**관찰**: 세 축에 안 속하는 "축 무관" 항목이 절반 가까이(8/16)다 — 이건 나쁜 신호가 아니라, 이
앱이 실제로 "정리·실행·통찰"이라는 사고 도구 위에 "파일 관리자"라는 인프라를 얹은 구조라는 뜻이다.
사용자가 지적한 "계층이 정리 안 된 느낌"은 **정리/실행/통찰 세 축 안에서** 발생했지, 인프라 쪽에서
발생한 게 아니다 — 그러니 이번 재편은 노드 팝업·선택 툴바·사이드바 상단 내비게이션에 집중하고
파일 IO·테마 같은 인프라는 건드리지 않는다.

## 3. 발견한 것 — 코드가 이미 알고 있던 것과 몰랐던 것

- 완료(done)와 일정(scheduled)은 코드 그룹핑에서 **이미** 분리돼 있었다(`ContextMenu.tsx`의 "속성"
  vs "일정 · 집중" 그룹) — 즉 "todo=완료"라는 모델이 암묵적으로 이미 있었지만, 완료가 색·아이콘과
  묶여 있어 "완료가 일정·집중과 한 흐름"이라는 건 안 보였다.
- 집중 *세션*(시간 추적)과 집중 *필터*(구 "노드에 집중" 뷰 전환)는 이미 결정 0006에서 이름을 분리해
  개념적으로 섞이지 않게 해뒀다 — 이번 재편은 그 위에 쌓는 것이지 되돌리는 게 아니다.
- 노트 시스템을 "통찰"로 분류하는 건 제품 한 줄 정의("생각(마인드맵), **기록**(노트), 할 일(일정·
  리마인더)")와도 맞다 — "기록"이 곧 "더 깊이 써서 이해를 남기는 것"이라는 해석과 일치한다.

## 4. UI 계층 재편 제안

### 4-1. 노드 우클릭 메뉴 — 5그룹 → 4그룹 (결정 0011에서 이미 확정)

| 기존 | 신규 |
|---|---|
| 편집(자식·형제·편집) | **구조**(자식·형제·편집·접기/펼치기·이 노드만 보기·확대·복제·삭제 — 편집+보기 그룹 통합) |
| 속성(색·아이콘·완료) | **정리**(색·아이콘·메모·링크) — 완료는 아래로 이동 |
| 첨부(메모·링크·노트연결) | **실행**(완료·일정·집중) — 신설, 이 흐름이 메뉴에서 처음으로 하나로 보임 |
| 일정 · 집중 | **통찰**(노드에 노트 연결) — "첨부"에서 분리, 통찰로 들어가는 다리로 재정의 |
| 보기 | (구조로 흡수) |

메뉴에 뜨는 **순서**도 축의 인과 순서(정리→실행→통찰)를 따른다: 구조 → 정리 → 실행 → 통찰.

### 4-2. 선택 툴바 — 순서·시각 그룹 재정렬

현재 6개 버튼(색상·완료·메모·집중·자식추가·삭제)은 정리/실행/구조가 뒤섞인 순서다. 축 순서로
재배열: `[색상 메모]`(정리) `|` `[완료 집중]`(실행) `|` `[자식추가 삭제]`(구조). 버튼 구성은 그대로,
순서와 구분선(`.st-sep`) 위치만 조정 — REDESIGN-VISION §T2의 "자주 쓰는 3~4개" 목표와 별개로, 있는
6개를 축 기준으로 다시 줄세우는 것.

### 4-3. 일정 → 집중을 하나의 흐름으로 연결 (결정 0011 §3)

`SchedulePopover`에 **"지금 집중 시작"** 액션을 추가한다 — 일정을 정한 바로 그 화면에서 실행까지
이어지도록. 이게 이번 재편에서 가장 눈에 띄는 변화가 될 것: 지금까지 완전히 분리된 화면(칩 클릭 →
날짜 카드 / 우클릭 → 별도 메뉴 항목)이던 두 동작이 한 카드 안에서 이어진다. 선택 툴바의 독립
"집중" 아이콘은 유지하되(G3 — 일정 없이도 기술적으로 가능해야 함), 우클릭 메뉴의 "실행" 그룹
안에서 일정 바로 아래에 놓아 "이 흐름의 다음 단계"라는 인상을 준다.

### 4-4. 사이드바 상단 내비게이션 재검토

현재 "오늘 · 집중 기록 · 최근 수정 · 즐겨찾기"가 한 블록으로 나열돼 있다. 축으로 보면:
- **오늘**: 오늘 일정이 있는 노드 모아보기 → **실행**(오늘 뭘 해내야 하는지)
- **집중 기록**: 대시보드/돌아보기 → **통찰**
- **최근 수정 · 즐겨찾기**: 파일 탐색 유틸리티 → 축 무관

셋을 한 줄에 나열하면 "오늘"과 "집중 기록"이 실행→통찰로 이어지는 짝이라는 게 "최근 수정/즐겨찾기"
때문에 묻힌다. 제안: **"오늘" + "집중 기록"을 위쪽에 묶고, 아래에 옅은 구분선, 그 아래 "최근 수정" +
"즐겨찾기"** — 새 UI 컴포넌트 없이 순서와 여백만 조정.

### 4-5. (검토했으나 반영 안 함) 노드 시각 표시에 축 반영

"노드가 지금 어느 축에 있는지"(정리 중/실행 중/통찰 연결됨)를 캔버스에서 색이나 아이콘으로 표시하는
안도 검토했으나 기각 — `color`는 이미 사용자 태깅용으로 예약돼 있고(결정 0001), 축 표시를 위한 새
시맨틱 색을 추가하면 3계층 색 시스템(`docs/design/COLOR-SYSTEM.md`)에 4번째 개념을 얹는 것이라
과할 위험이 크다. 기존 게이지 칩(일정=주황, 노트=보라)이 이미 "이 노드가 실행/통찰과 연결돼 있다"는
신호 역할을 하고 있으므로 이걸로 충분하다고 판단.

## 5. 범위 (포함 / 제외)

**포함**: 노드 우클릭 메뉴 재편(4-1), 선택 툴바 재정렬(4-2), SchedulePopover↔집중 연결(4-3), 사이드바
상단 내비게이션 순서·여백 조정(4-4).

**제외**:
- 다중 선택 메뉴는 축 재편 대상에서 제외 — 다중 선택은 "구조"(연결·섹션)와 "속성"(색·완료) 2그룹뿐
  이라 이미 단순하고, 여러 노드에 걸친 "실행"·"통찰" 일괄 동작(예: 여러 노드 한번에 일정 걸기)은
  범위 밖 신규 기능이라 이번엔 안 다룬다.
- 노드 시각 표시 축 반영(4-5) — 위 사유로 기각.
- 검색·커맨드 팔레트의 축별 재구성 — 세 축을 가로지르는 유틸리티라 강제로 나누면 오히려 찾기 어려워짐.

## 6. 위험

- **재분류가 사용자의 실제 멘탈모델과 다를 수 있다** — 특히 메모/링크를 "정리"로, 노트연결만
  "통찰"로 가른 것은 사용자 확인이 필요한 판단 지점이다(§7 열린 질문 1).
- 우클릭 메뉴 그룹 이름이 "편집/속성/첨부/일정 · 집중/보기"에서 "구조/정리/실행/통찰"로 바뀌면,
  이 이름 자체가 새로운 내부 용어가 될 위험이 있다 — UX-CLARITY-VISION 전략 F(사용자 언어로 번역)
  원칙과 충돌하지 않는지 확인 필요. "정리"·"실행"·"통찰"은 이미 일상어라 큰 위험은 아니라고 판단.
- SchedulePopover에 집중 시작 버튼을 추가하면 그 카드가 커진다 — 정보 밀도 원칙(모달 380–420px,
  카드 한 줄 기본)과 균형 확인 필요.

## 7. 열린 질문 (사용자 확인 필요)

1. **메모·링크를 "정리"로 분류한 것에 동의하시나요?** 아니면 이 둘도(특히 링크는 외부 자료 참조라는
   점에서) "통찰"에 가깝다고 보시나요?
2. **선택 툴바의 독립 "집중" 아이콘을 완전히 없애고 SchedulePopover 경유만 남길지, 아니면 이번
   제안대로 백업 경로로 유지할지?** (G3 준수 차원에서 유지를 제안했지만, 더 강하게 "일정 우선"을
   원하시면 제거도 가능)
3. 사이드바 "오늘"+"집중 기록" 묶음, "최근 수정"+"즐겨찾기" 묶음으로 나누는 것 외에, 두 그룹 사이에
   시각적 구분(옅은 배경 등)까지 원하시는지, 아니면 순서·여백 조정 정도로 충분한지?

승인되는 대로 구현 순서는: 4-1(우클릭 메뉴, 위험 낮음) → 4-2(선택 툴바, 위험 낮음) → 4-4(사이드바,
위험 낮음) → 4-3(SchedulePopover 확장, 카드 레이아웃 변경 필요해 위험 중간)을 제안한다.
