# 기능 계층 재정의 — 정리 · 실행 · 통찰 3축

> 작성: 2026-07-15 · 갱신: 2026-07-15 (사용자 확인 반영, 구현 진행) · 상태: **확정 — 구현 중**
> 근거: [결정 0011](../decisions/0011-three-axis-thinking-model.md), 사용자가 직접 제시한 3축 재정의.
> 관계: [UX-CLARITY-VISION-2026-07.md](UX-CLARITY-VISION-2026-07.md)를 대체하지 않는다 — 그 문서는
> "같은 기능이 어디서 열든 같은 모양이어야 한다"(시각적 일관성)였고, 이 문서는 "기능이 애초에 왜 이
> 그룹에 속하는가"(개념적 계층)다. UX-CLARITY가 만든 공유 컴포넌트(ColorSwatchGrid, useDismissablePosition
> 등)는 그대로 재사용한다 — 이 문서는 그 위에 얹는 재배치 작업.

## 0. 왜 다시 여는가

지난 UX-CLARITY-VISION 작업은 "우클릭 메뉴/선택 툴바/색 선택기가 어디서 열려도 같은 모양"이라는
**시각적 일관성**을 만들었다. 하지만 사용자가 지적한 문제는 그보다 한 겹 아래에 있다 — **"완료·일정·
집중이 왜 서로 다른 그룹에 있는가", "메모·링크·노트연결을 '첨부'로 묶은 게 맞는가"** 같은, 그룹 자체의
존재 이유에 대한 질문이다. 시각적으로 아무리 일관돼도, 그룹핑 기준 자체가 사용자의 실제 사고 흐름과
안 맞으면 "정리가 안 된 느낌"은 사라지지 않는다.

## 1. 세 축 (요약 — 전체 정의는 [결정 0011](../decisions/0011-three-axis-thinking-model.md))

- **정리(Organize)** — 생각을 포착·구조화
- **실행(Execute)** — 정리된 것 중 할 일을 실제로 해냄 (완료→일정→집중이 하나의 흐름)
- **통찰(Insight)** — 더 깊이 쓰고 되짚어 이해

## 2. 전체 기능 재평가 — 축 매핑

`docs/product/FEATURE-INVENTORY.md`의 16개 영역을 축 기준으로 재분류한다. "축 무관"은 세 축
어디에도 속하지 않는 **인프라/내비게이션** — 파일을 열고 검색하고 창을 나누는 것 같은, 사고 자체가
아니라 사고를 담는 그릇에 해당한다.

**관찰**: 세 축에 안 속하는 "축 무관" 항목이 절반 가까이(8/16)다 — 이건 나쁜 신호가 아니라, 이
앱이 실제로 "정리·실행·통찰"이라는 사고 도구 위에 "파일 관리자"라는 인프라를 얹은 구조라는 뜻이다.
사용자가 지적한 "계층이 정리 안 된 느낌"은 **정리/실행/통찰 세 축 안에서** 발생했지, 인프라 쪽에서
발생한 게 아니다 — 그러니 이번 재편은 노드 팝업·선택 툴바·사이드바 상단 내비게이션에 집중하고
파일 IO·테마 같은 인프라는 건드리지 않는다.

## 3. 발견한 것 — 코드가 이미 알고 있던 것과 몰랐던 것

- 완료(done)와 일정(scheduled)은 코드 그룹핑에서 **이미** 분리돼 있었다(`ContextMenu.tsx`의 "속성"
  vs "일정 · 집중" 그룹) — 즉 "todo=완료"라는 모델이 암묵적으로 이미 있었지만, 완료가 색·아이콘과
  묶여 있어 "완료가 일정·집중과 한 흐름"이라는 건 안 보였다.
- 집중 *세션*(시간 추적)과 집중 *필터*(구 "노드에 집중" 뷰 전환)는 이미 결정 0006에서 이름을 분리해
  개념적으로 섞이지 않게 해뒀다 — 이번 재편은 그 위에 쌓는 것이지 되돌리는 게 아니다.
- 노트 시스템을 "통찰"로 분류하는 건 제품 한 줄 정의("생각(마인드맵), **기록**(노트), 할 일(일정·
  리마인더)")와도 맞다 — "기록"이 곧 "더 깊이 써서 이해를 남기는 것"이라는 해석과 일치한다.

## 4. UI 계층 재편 제안

### 4-1. 노드 우클릭 메뉴 — 5그룹 → 4그룹 (결정 0011 §4, 확정)

| 기존 | 신규 |
|---|---|
| 편집(자식·형제·편집) | **구조**(자식·형제·편집·접기/펼치기·이 노드로 확대·복제·삭제) |
| 속성(색·아이콘·완료) | **정리**(색·아이콘·메모·링크) — 완료는 아래로 이동 |
| 첨부(메모·링크·노트연결) | **실행**(완료·일정·집중 — 집중은 일정 있을 때만 활성) |
| 일정 · 집중 | **통찰**(이 노드만 보기, 노드에 노트 연결) |
| 보기 | (구조/통찰로 분산 — 확대는 구조, 이 노드만 보기는 통찰) |

메뉴에 뜨는 **순서**도 축의 인과 순서(정리→실행→통찰)를 따른다: 구조 → 정리 → 실행 → 통찰.

### 4-2. 선택 툴바 — 순서·시각 그룹 재정렬 + 집중 버튼 조건부 표시

현재 6개 버튼(색상·완료·메모·집중·자식추가·삭제)은 정리/실행/구조가 뒤섞인 순서다. 축 순서로
재배열: `[색상 메모]`(정리) `|` `[완료 집중]`(실행) `|` `[자식추가 삭제]`(구조). **집중 버튼은
`node.scheduled`일 때만 렌더링**(확정, 아래 4-3) — 일정 없는 노드를 선택하면 툴바에 5개만 보이고,
일정 있는 노드를 선택하면 집중 버튼이 나타난다. 이 등장/소실 자체가 "일정이 있어야 집중이 가능하다"
는 신호가 된다.

### 4-3. 집중은 일정이 지정된 노드에서만 (확정, 결정 0011 §3)

- **선택 툴바**: `node.scheduled`가 아니면 집중 버튼 자체를 렌더링하지 않는다(4-2).
- **우클릭 메뉴**: "집중 세션 시작" 항목은 항상 보이되, 일정이 없으면 비활성(disabled) + 안내
  문구("일정 설정 후 이용 가능")를 붙인다 — 완전히 숨기지 않는 이유는 존재 자체를 알 수 있어야
  하기 때문(U1). 일정이 있으면 활성 상태로 정상 동작.
- **SchedulePopover**: **"지금 집중 시작"** 버튼 신설 — 일정을 정한 바로 그 화면에서 실행까지
  이어지는, 가장 눈에 띄는 진입점. 날짜/시간 입력·미리알림 토글 아래, "스케줄 지우기" 위에 배치.
  primary 스타일로 강조(카드 안에서 유일하게 색이 있는 액션 버튼).

### 4-4. 사이드바 재편 — 캘린더 탭 신설 + 스마트 뷰 재배치 (확정, 신규 결정 2026-07-15)

"오늘"을 정적인 오버레이 목록에서 **캘린더 탭**으로 승격한다 — 일간·주간·월간 보기가 가능한 진짜
캘린더. 상세 명세는 [specs/2026-07-15-calendar-view.md](specs/2026-07-15-calendar-view.md), 구현
계획은 [exec-plans/completed/2026-07-15-calendar-view.md](../exec-plans/completed/2026-07-15-calendar-view.md)
(구현 완료).

**사이드바 상단 내비게이션**: "오늘 · 집중 기록 · 최근 수정 · 즐겨찾기" 4개 → **"캘린더 · 즐겨찾기"**
2개만 남긴다.

- **캘린더**: 클릭하면 탭으로 열림(오버레이 아님 — 다른 탭처럼 닫기/전환 가능). 첫 화면은 오늘 날짜의
  일정이 우선 보이는 상태(일간 뷰 기본, 오늘 날짜로 스크롤/포커스). 일간·주간·월간 전환 가능.
- **즐겨찾기**: 기존 그대로 유지.

**이동한 것**:
- **최근 수정**: 사이드바에서 빠지고 **설정(⌘,) 안으로 이동** — 사용자 지시("최근 수정도 설정
  내부로 옮겨도 좋아").
- **집중 기록(대시보드/돌아보기)**: 사용자가 "판단해서 정하라"고 위임한 항목. 두 경로 모두 제공하기로
  결정:
  1. **캘린더 탭 안에서 접근** — 캘린더 헤더에 "돌아보기" 아이콘 버튼(또는 각 날짜 셀에 그날의
     집중 기록이 있으면 작은 표시 + 클릭 시 대시보드) — 일정(실행)을 보다가 바로 그 회고(통찰)로
     넘어가는 자연스러운 동선이라 채택.
  2. **사이드바 맨 아래(하단 바)** — 휴지통·노트 템플릿과 같은 자리에 아이콘 추가. 근거: 대시보드는
     "관리 화면"류(0007에서 이미 "관리형 화면 아이콘 묶음"으로 분류된 휴지통·Note Template과 같은
     성격)이지 매일 들여다보는 1차 내비게이션이 아니다 — 설정보다 하단 바가 더 어울린다(설정은
     "환경을 바꾸는 곳"이지 "콘텐츠를 보는 곳"이 아니라서). **사이드바 맨 아래를 선택.**

결과적으로 사이드바 상단은 캘린더·즐겨찾기 2줄로, 하단 바는 휴지통·Note Template·**집중 기록**·설정
4개가 된다.

### 4-5. (검토했으나 반영 안 함) 노드 시각 표시에 축 반영

"노드가 지금 어느 축에 있는지"(정리 중/실행 중/통찰 연결됨)를 캔버스에서 색이나 아이콘으로 표시하는
안도 검토했으나 기각 — `color`는 이미 사용자 태깅용으로 예약돼 있고(결정 0001), 축 표시를 위한 새
시맨틱 색을 추가하면 3계층 색 시스템(`docs/design/COLOR-SYSTEM.md`)에 4번째 개념을 얹는 것이라
과할 위험이 크다. 기존 게이지 칩(일정=주황, 노트=보라)이 이미 "이 노드가 실행/통찰과 연결돼 있다"는
신호 역할을 하고 있으므로 이걸로 충분하다고 판단.

## 5. 범위 (포함 / 제외)

**포함**: 노드 우클릭 메뉴 재편(4-1), 선택 툴바 재정렬 + 집중 조건부 표시(4-2, 4-3), SchedulePopover
"지금 집중 시작"(4-3), 캘린더 탭 신설(4-4, 별도 명세/계획), 사이드바 재편(4-4).

**제외**:
- 다중 선택 메뉴는 축 재편 대상에서 제외 — 다중 선택은 "구조"(연결·섹션)와 "속성"(색·완료) 2그룹뿐
  이라 이미 단순하고, 여러 노드에 걸친 "실행"·"통찰" 일괄 동작(예: 여러 노드 한번에 일정 걸기)은
  범위 밖 신규 기능이라 이번엔 안 다룬다.
- 노드 시각 표시 축 반영(4-5) — 위 사유로 기각.
- 검색·커맨드 팔레트의 축별 재구성 — 세 축을 가로지르는 유틸리티라 강제로 나누면 오히려 찾기 어려워짐.
- 캘린더에서 드래그로 일정 재조정, 반복 일정 — 캘린더 v1 비목표([명세](specs/2026-07-15-calendar-view.md) §3 참고).

## 6. 위험

- 우클릭 메뉴 그룹 이름이 "편집/속성/첨부/일정 · 집중/보기"에서 "구조/정리/실행/통찰"로 바뀌면,
  이 이름 자체가 새로운 내부 용어가 될 위험이 있다 — UX-CLARITY-VISION 전략 F(사용자 언어로 번역)
  원칙과 충돌하지 않는지 확인 필요. "정리"·"실행"·"통찰"은 이미 일상어라 큰 위험은 아니라고 판단.
- SchedulePopover에 집중 시작 버튼을 추가하면 그 카드가 커진다 — 정보 밀도 원칙(모달 380–420px,
  카드 한 줄 기본)과 균형 확인 필요.
- 캘린더 탭은 워크스페이스 전체(모든 `.mind` 파일)를 스캔해 일정 있는 노드를 모아야 한다 —
  `GlobalSearch`가 이미 하는 전체 스캔 패턴을 재사용하되, 맵이 많으면 느려질 수 있어 성능 확인 필요.

## 7. 확정된 결정 (사용자 확인 완료 — 2026-07-15)

1. **메모·링크 = 정리.** (확정)
2. **이 노드만 보기 = 통찰**, **메타 블록(정보 양식) = 정리.** (확정, §2 테이블 갱신 반영)
3. **집중은 일정이 지정된 노드에서만 나온다** — 선택 툴바는 조건부 렌더링, 우클릭 메뉴는 비활성+안내.
   SchedulePopover에 "지금 집중 시작" 신설. (확정, §4-3)
4. **사이드바는 캘린더·즐겨찾기 2개만** — 캘린더는 일/주/월 뷰를 갖춘 탭, 첫 화면은 오늘 일정 우선.
   최근 수정은 설정으로, 집중 기록(대시보드)은 "캘린더 안 경로 + 사이드바 하단 바" 둘 다로 이동
   (판단 위임 항목 — §4-4에서 근거와 함께 결정). (확정, §4-4)

구현 순서: 4-1(우클릭 메뉴) → 4-2/4-3(선택 툴바+집중 게이트+SchedulePopover) → 사이드바 재편(최근
수정 이동, 상단 내비 축소) → 캘린더 탭(가장 큰 작업, 별도 명세/계획 참고).
