# 정보구조 전략 — Light/Heavy 사용자 간극을 메우는 기능 계층화

> 작성: 2026-07-10 · 상태: **§5-1/5-2/5-3(A)/5-5/5-6 구현 완료 (2026-07-10) · 5-4는 AI 기획 확정 대기, 6b는 별도 명세 필요**
> 근거: [PRODUCT-DEFINITION.md](PRODUCT-DEFINITION.md)(타깃·비목표), [FEATURE-INVENTORY.md](FEATURE-INVENTORY.md)(전체 기능 15종 전수), [decisions/0004](../decisions/0004-entrypoint-matrix.md)(진입점 레이어 매트릭스, 이미 합의됨), [specs/2026-06-11-entrypoint-matrix.md](specs/2026-06-11-entrypoint-matrix.md)(보류 항목)
> 관련: [note-templates.md](specs/2026-07-09-note-templates.md) — 이 문서를 쓰게 된 직접 계기(메타+/템플릿+ 혼동 신고)

## 0. 이 문서가 새로 하는 것 / 이미 있는 것 위에 얹는 것

이 저장소엔 이미 **"기능 하나가 어떤 진입점들을 갖는가"**를 규정하는 문서가 있다 — 진입점 레이어 매트릭스(ADR 0004): 단축키(고빈도 전용) / 우클릭 메뉴(전체 카탈로그) / 선택 툴바(6±2개) / 노드 팝오버(상세 편집) / 메뉴바(앱 수준) / ⌘K(전체 미러). 이 매트릭스는 잘 작동하고 있고 건드리지 않는다.

이 매트릭스가 **규정하지 않는 것**이 하나 있다: **이 기능을 애초에 사용자에게 언제 보여줄 것인가.** 지금은 "기능이 만들어진 순서"가 곧 "노출 순서"다 — 새 기능이 나올 때마다 사이드바 푸터나 툴바에 아이콘이 하나씩 늘어나는 식. 그 결과 지금 앱을 처음 여는 사람과 1년 써온 사람이 **완전히 같은 화면**을 본다. 이 문서는 그 위에 **"빈도 기반 노출 계층"**이라는 새 축을 하나 추가한다.

## 1. 근거 — 지금 실제로 무슨 일이 일어나고 있는가

`FEATURE-INVENTORY.md` 전수 조사 결과:

- **검색 진입점이 3개** (`⌘F` 노드 검색 / `⌘P` 파일명 검색 / `⌘⇧F` 전체 검색) — 이름·단축키·범위가 서로 겹쳐 보여서 어떤 상황에 뭘 눌러야 하는지 기억해야 한다. `FEATURE-INVENTORY.md` 자체도 "전역 검색 없음(로드맵 후보)"이라고 잘못 적혀 있었다 — 문서를 쓴 사람조차 이미 3개나 있다는 걸 잊을 만큼 진입점이 흩어져 있었다는 방증.
- **설정의 "AI 기능" 섹션은 지금은 인프라뿐이지만 곧 핵심 확장 축이 된다** (2026-07-10 방향 전환, [PRODUCT-DEFINITION.md](PRODUCT-DEFINITION.md) 갱신 완료) — 이전 초안은 이걸 "쓰는 사람 없는 죽은 UI"로 보고 삭제를 제안했으나 **철회**했다. 대신 남는 질문은 "삭제할지"가 아니라 "AI 기능이 구체화되면 어느 Tier·어느 진입점에 놓을지"다(§5-4, §6 열린 질문 3번 참조).
- **"메타+"와 "템플릿+"가 툴바에 나란히 있다** — 완전히 다른 기능(구조화 필드 vs 본문 텍스트 삽입)인데 이름·모양·위치가 거의 동일해서, 이 대화에서 실제로 사용자가 헷갈려 질문했다. 새 기능(템플릿+)이 기존 기능(메타+) 옆에 "일단 비슷하게" 얹힌 전형적 사례.
- **우클릭 메뉴 14항목 그룹핑이 5주째 미결** (ADR 0004, "보류") — 그룹핑 여부를 아직도 못 정한 이유 자체가, 계층화 기준이 없어서다.
- **사이드바 푸터가 4개 아이콘**(집중 타이머 · 휴지통 · Note Template · 설정)로 늘어났다 — 템플릿 기능(2026-07) 하나가 배포될 때마다 여기 하나씩 늘어나는 패턴이 이미 시작됨.

이건 "기능이 나빠서"가 아니라 **성장 방식이 나빠서**다 — 매번 "이번 기능을 어디 붙일까"만 결정했지, "이 기능이 몇 % 사용자에게 며칠 차에 필요한가"를 물은 적이 없다.

## 2. 두 페르소나 — `PRODUCT-DEFINITION.md`의 타깃을 구체화

제품 정의는 타깃을 "개인 지식 작업자"로 뭉뚱그렸지만, 실제로는 **같은 타깃 안에 사용 강도가 다른 두 층**이 있고 이 둘의 간극이 지금 IA 혼란의 근본 원인이다.

### Light — "빠른 캡처형"
- 일주일에 2~3번, 세션 10~20분. 맵 1~3개, 노트는 가끔.
- 원하는 것: 생각나는 대로 노드 치기, 색으로 대충 분류, 필요하면 메모 한 줄. **저장이 알아서 되는 것**만으로 이미 만족.
- 지금 화면에서 이들을 방해하는 것: "오늘"·"집중 기록" 스마트 네비(안 씀), 리마인더 동기화 설정(안 씀), 메타 템플릿 스키마 빌더(존재조차 모름), 템플릿+/메타+ 두 버튼의 차이(헷갈림), 검색 3종 중 뭘 눌러야 할지.

### Heavy — "지식 운영형"
- 매일, 여러 시간. 맵·노트 수십 개, 분할 뷰 상시 사용.
- 원하는 것: 노드=일정=리마인더 왕복 신뢰성, 프로젝트별 메타 스키마, 회의록 등 반복 노트 템플릿, 파일 전체를 가로지르는 검색, 집중 시간 기록.
- 이들에게도 지금 상태가 나쁜 이유: 정작 필요한 리마인더 동기화·전역 검색이 방향 미확정인 AI 설정, 메타/템플릿 구분 문제와 **같은 우선순위로** 뒤섞여 있어서, 매번 "이게 그 기능 맞나"를 재확인해야 한다. 카탈로그(우클릭 메뉴)가 길어지는 것 자체는 이들에게 문제가 아니다 — 오히려 다른 곳(툴바·설정)까지 카탈로그처럼 평평해지는 게 문제.

**간극**: Light 사용자에게 필요한 건 "적게 보이는 것"이고, Heavy 사용자에게 필요한 건 "다 있되 헷갈리지 않는 것"이다. 이 둘은 상충하지 않는다 — **기본값을 얕게, 그러나 얕은 화면에서 깊은 화면으로 가는 다리를 명확히** 놓으면 둘 다 만족된다. 지금 문제는 "너무 많이 보여서"가 아니라 **"얕은 계층과 깊은 계층이 뒤섞여서"** 다.

## 3. 원칙: Depth = 필요 빈도, 다리(Bridge) = 성장 트리거

```
Tier 0 · Core        │ 첫 세션에 필요, 항상 1클릭/기본 단축키
                      │ (노드 CRUD, 색, 저장, 기본 노트 1개 연결, 단일 검색 진입점)
──────────────────────┼─────────────────────────────────────
Tier 1 · Grows-with   │ 1~2주차에 필요해짐, "한 걸음 더" 위치
                      │ (분할 뷰, 섹션/연결선, 집중 세션, 메타/노트 템플릿 *사용*)
──────────────────────┼─────────────────────────────────────
Tier 2 · Power/Setup  │ 의도적으로 찾아야 함, 설정 하위 화면 또는 "고급"
                      │ (리마인더 동기화 *설정*, 메타 템플릿 *스키마 관리*,
                      │  노트 템플릿 *관리*, AI 프로바이더 설정 — Tier는 기획 확정 후 재검토)
```

- **Tier가 낮다고 기능이 열등한 게 아니다** — Tier 2는 "매일 안 건드리는 설정"이지 "덜 중요한 기능"이 아니다. 리마인더 동기화는 제품의 핵심 차별 기능(§PRODUCT-DEFINITION)이지만, **"켜는 것"은 Tier 2**(한 번 설정), **"쓰는 것"(노드에 날짜 걸기)은 Tier 0/1**(선택 툴바·팝오버, 이미 그렇게 돼 있음 — 올바른 예).
- **레이어 매트릭스(ADR 0004)는 그대로 유지, Tier는 그 위에 별도 축으로 얹는다.** 예: 템플릿+ 버튼은 "툴바"라는 레이어에 있으면서(매트릭스), 동시에 Tier 1(그로우) 판정 — 즉 툴바에 상시 노출하되, **첫 노트 3개 작성 전까진 접힌 상태**로 두는 식(§5 참조)으로 두 축을 조합한다.
- **다리(성장 트리거)**: 얕은 계층에서 깊은 계층으로 자연스럽게 건너가는 조건을 명시한다. 이게 없으면 Tier 2는 영원히 발견 안 되는 죽은 기능이 된다.

| 트리거 조건 | 넛지 |
|---|---|
| 한 워크스페이스에 노드 날짜(`scheduled`)가 3개 이상 생김 | "리마인더 앱과 동기화해서 놓치지 않게 할까요?" (설정 딥링크) |
| 비슷한 제목의 노트를 3개 이상 만듦(예: "회의록 …") | "이 구조를 템플릿으로 저장해둘까요?" |
| 같은 워크스페이스에서 탭을 5개 이상 동시에 엶 | 분할 뷰(`⌘K` 화면 분할) 힌트 1회 노출 |
| 노트에 구조화된 값(날짜·URL 등)을 반복 입력 | 메타 템플릿 소개(1회) |

## 5. 우선순위 높은 구체적 제안 (임팩트 순)

### 5-1. 검색 3종 → 하나의 진입점으로 통합 (최우선)

- **문제**: `⌘F`(맵 내부 노드) / `⌘P`(파일명) / `⌘⇧F`(전체 내용)가 이름·단축키가 다 비슷한데 범위가 다 다르다. Light 사용자는 이 셋을 구분할 이유가 없고, Heavy 사용자도 "어느 걸 눌러야 하지"를 매번 재확인한다.
- **제안**: `⌘K`류 하나의 오버레이로 합치되, **범위를 사용자가 고르지 않고 맥락으로 자동 결정**한다 — 맵이 열려 있으면 기본은 "이 맵"(구 `⌘F` 동작), 입력을 시작하자마자 결과 하단에 "전체에서 찾기 →"(구 `⌘⇧F`)를 얕은 확장으로 둔다. 파일 자체를 열려는 의도(구 `⌘P`)는 같은 입력창에서 파일 아이콘이 붙은 결과로 자연 통합(이미 `CommandPalette.tsx`가 부분적으로 이렇게 하고 있음 — 그 패턴을 검색까지 확장).
- **레이어 매트릭스 영향**: ⌘K 레이어의 역할("모든 명령의 검색 가능한 합집합")을 검색까지 문자 그대로 확장하는 것 — 매트릭스를 어기지 않고 오히려 완성한다.
- **왜 최우선인가**: 셋 다 "찾기"라는 같은 사용자 의도를 놓고 진입점만 3개인, 이 문서에서 발견한 가장 순수한 "분산" 사례. 삭제되는 기능이 없어 리스크가 가장 낮다.

> **2026-07-10 구현 완료 (범위를 좁혀서)**: `⌘F`/`⌘P`/`⌘⇧F` 세 컴포넌트를 하나로 합치는 전면 재작성은 하지 않았다 — 각자 독립 구현·서로 다른 성능 특성(`GlobalSearch`는 열 때마다 워크스페이스 전체를 다시 읽음)이라 리스크가 컸다. 대신:
> - `CommandPalette`(⌘K)가 활성 맵이 있으면 노드 텍스트도 함께 검색해 "노드" 그룹으로 보여주고(`src/ui/CommandPalette.tsx`), 쿼리가 있을 때 항상 "전체에서 찾기 (⌘⇧F)" 힌트를 하단에 노출 — 클릭하면 같은 쿼리를 이어받아 `GlobalSearch`를 연다.
> - `QuickOpen`(⌘P)도 파일명으로 못 찾으면 빈 상태에 같은 탈출구를 보여준다.
> - `⌘F`/`⌘P`/`⌘⇧F` 자체는 전혀 건드리지 않았다 — 이미 익숙한 사용자의 단축키는 그대로, ⌘K가 "모르면 일단 여기" 역할을 하도록 만든 것. E2E: `e2e/ia-consolidation.spec.ts`.

### 5-2. 사이드바 푸터 재정리

- 현재: 집중 타이머 · 휴지통 · Note Template · 설정, 4개 아이콘. 템플릿 기능 하나가 배포될 때마다 여기 하나씩 늘어나는 패턴이 이미 시작됨 — 다음 기능이 나오면 5개, 6개가 된다.
- **제안**: 휴지통과 Note Template 관리(15b)는 둘 다 "가끔 열어보는 관리 화면"이라는 같은 역할이다. 아이콘 2개를 유지하되(발견성은 지키고), **"라이브러리 관리" 같은 상위 그룹 레이블 없이 시각적으로만 묶어** 향후 3번째 관리 화면이 추가돼도 푸터가 또 늘어나지 않게 확장 여지를 만든다. 새 "관리형" 기능은 기본적으로 여기 묶음에 편입하는 것을 규칙으로 삼는다(향후 새 기능마다 별도 아이콘 신설 금지 — 매트릭스에 6번째 레이어를 새로 만들지 않는다).

> **2026-07-10 구현 완료**: `.sb-foot-group`로 휴지통·Note Template을 옅은 배경 묶음으로, 설정 버튼은 분리 (`src/sidebar/Sidebar.tsx`, `src/styles.css`). 새 관리 화면은 이 묶음에 편입하는 게 기본 규칙.

### 5-3. 메타+ / 템플릿+ 관계를 명확히 한다

- 이 대화에서 실제로 벌어진 사례: 완전히 다른 데이터 모델(구조화 필드 vs 본문 텍스트)인데 버튼 이름·모양·위치가 "X+"로 통일돼 있어 헷갈렸다.
- **두 가지 옵션, 팀 결정 필요**:
  - **(A) 시각적으로 더 벌린다** — 이름을 "메타 필드"/"템플릿 삽입"처럼 겹치지 않게 바꾸고, 색(형태)으로 이미 구분되고 있는 `--state-note` 재사용을 유지하되 아이콘 모양 차이를 더 키운다. 가장 작은 변경.
  - **(B) 하나의 "삽입" 메뉴로 통합한다** — 사용자 멘탈 모델에서 둘 다 "노트에 뭔가를 끼워 넣는다"이므로, 버튼 하나("＋ 삽입") 안에 "메타 필드"/"템플릿" 두 섹션으로 나눠 넣는다. 버튼 수는 줄지만 구현 변경이 크다.
  - 이 문서는 (A)를 최소 리스크안으로 권장하되, 최종 선택은 팀 논의(§6 참조).
- 둘 다 **Tier 1**(그로우)로 재분류 제안: 툴바에 상시 노출하되, "첫 노트 3개 작성 전"에는 접혀 있다가(§3 트리거) 자연스럽게 펼쳐지는 걸 검토— 첫 화면을 더 비우는 효과.

> **2026-07-10 구현 완료 — (A) 채택**: 두 버튼 모두 아이콘을 추가해 라벨이 같은 패턴("X+")이어도 모양으로 구분된다 — 메타+는 `table` 아이콘(`--primary` 파랑, 기존 설정 화면의 "메타 템플릿" 표기와 동일 색), 템플릿+는 `template` 아이콘(`--state-note` 보라, 템플릿 정체성 색 재사용). 라벨 문구·위치는 안 바꿨다(이미 익숙한 사용자의 muscle memory 보존). (B) 통합 버튼안은 채택하지 않음 — §6 열린 질문 1번 해소.

### 5-4. AI 프로바이더 설정 — 삭제 제안 철회, 재배치는 기획 확정 후

> **2026-07-10 정정**: 이 절의 최초 버전은 "AI 기능"이 `PRODUCT-DEFINITION.md`의 v1 비목표였다는 근거로 설정 화면에서 삭제를 제안했다. 그 사이 제품 방향이 AI를 **핵심 확장 축**으로 삼는 쪽으로 바뀌었고, `PRODUCT-DEFINITION.md`도 갱신했다 — 삭제 제안은 철회한다.

- 지금 있는 Claude/OpenAI 키 저장·전환 인프라(FEATURE-INVENTORY #14)는 그대로 둔다. 코드도, 설정 화면 위치도 건드리지 않는다.
- **남는 질문**은 "삭제할지"가 아니라 "AI 기능이 무엇을 하는지 정해지면 어느 Tier·어느 진입점에 놓을지"다 — 예를 들어 노트 요약처럼 Light 사용자도 첫 주에 쓸 만한 기능이면 Tier 0/1(툴바·팝오버)에 걸리고, 프로젝트 단위 대화형 기능이면 Tier 2(설정 하위)에 남을 수도 있다. **AI 기능의 실제 범위가 나오기 전엔 이 문서가 먼저 판단하지 않는다** — §6 열린 질문 3번 참조.

### 5-5. 우클릭 메뉴 14항목 — ADR 0004 보류 항목 마감

- 매트릭스 자체가 "우클릭 메뉴 = 전체 카탈로그" 역할을 이미 못 박았으므로, **길이 자체는 문제가 아니다** — Heavy 사용자에게 카탈로그가 길다고 나쁜 게 아니다. 남은 건 "스크롤 vs 구분선 그룹"이라는 순수 시각적 결정뿐.
- **제안**: 구분선 그룹(기존 "A안")으로 확정 — 편집/구조/속성/일정·리마인더/노트/집중/보기 7개 그룹, 그룹 내부 항목 수는 그대로. 새 기능(§4의 신설 항목 포함)이 메뉴에 들어올 때마다 이 7개 그룹 중 하나에 배정하는 걸 규칙화해서, 이 결정이 또 "보류"로 남지 않게 한다.

> **2026-07-10 구현 완료 (조사로 재확인)**: 실제로 조사해보니 단일 노드 메뉴는 이미 5개 그룹으로 레이블링돼 있었다(편집/속성/첨부/일정·집중/보기) — 문서 작성 시점의 "14항목 미그룹" 진단이 낡은 정보였다. 실제 공백은 **다중 선택 메뉴**(2개 이상 선택)에 그룹 레이블이 전혀 없었던 것 — "구조"(연결·섹션)/"속성"(색·완료) 레이블을 추가했다(`src/menu/ContextMenu.tsx`). 7그룹 재설계는 불필요.

### 5-6. 성장 트리거 넛지 — 최소 구현

- §3의 4개 트리거 조건을 토스트 1회성 넛지로 구현(닫으면 다시 안 뜸, `localStorage` 플래그). 새 UI 서피스를 만들지 않고 기존 `useUi.getState().toast()` 패턴 재사용 — 구현 비용 매우 낮음.

> **2026-07-10 구현 완료 (3/4 트리거)**: `src/ui/GrowthNudges.tsx` 신설, `useUi.getState().toastAction()` 재사용(이미 지원되던 패턴), `localStorage` 1회성 플래그는 기존 `focusCoachShown` 관례를 그대로 따름.

> - 제목이 비슷한 노트 3개 이상 → 템플릿 제안 (구현 — "제목 없음" 기본값 노트는 제외하도록 보정, E2E로 오탐 발견·수정)
> - 열린 맵의 스케줄 노드 3개 이상 → 리마인더 동기화 제안 (구현 — 워크스페이스 전체가 아니라 **현재 열린 맵**으로 범위 축소; 워크스페이스 전체를 스캔하려면 모든 `.mind` 파일을 다시 읽어야 해서 비용 대비 가치가 낮다고 판단)
